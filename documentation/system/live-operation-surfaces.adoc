= Live Operation Surfaces

Tiny Swarm World contains guarded Python workflows, transitional direct
operator scripts, retired historical surfaces, and supported assets. This
catalog is static repository documentation. It must be maintained without
executing live infrastructure commands.

== Classification Meanings

[options="header"]
|===
| Status | Meaning | Default quality-gate handling
| Supported | Current documented path for intentional operator use. Mutating paths still require the documented live-consent or operator review controls. | May be tested only with mocked or static checks unless the user explicitly requests live infrastructure execution.
| Transitional | Kept for compatibility while behavior moves behind workflow-level contracts. | Static review only by default; do not run during normal development.
| Retired | Removed as an executable entry point; retained only as historical documentation or navigation notes. | Assert absence of executable helpers and route operators to the Python setup workflow.
| Supported Asset | Configuration or image material consumed by supported or transitional workflows, not a standalone operator entry point. | Static validation only unless an explicit live deployment is requested.
|===

== Guarded Python Entrypoint

[options="header"]
|===
| Surface | Status | Notes
| `PYTHONPATH=src python3 -m tiny_swarm_world --list-workflows` | Supported | Lists workflow contracts without building application services.
| `PYTHONPATH=src python3 -m tiny_swarm_world --preflight` | Supported | Runs static preflight checks for the default `lxc_native` provider without live infrastructure execution.
| `PYTHONPATH=src python3 -m tiny_swarm_world platform verify` | Supported | Non-mutating workflow-level verification path. It runs post-install preflight and read-only checks for managed LXC nodes, Docker Engine readiness, Swarm membership, profile-level proxy devices, and the Portainer local endpoint.
| `platform init` | Supported | Mutating workflow name is supported and live-consent gated. The default path selects `lxc_native` through LXD or Incus using the explicit CLI backend, configured preferred backend, or ordered backend candidate list, ensures provider nodes, then runs Docker Engine setup and Swarm bootstrap inside the managed LXC nodes. It blocks before mutation when provider readiness, WSL2 capability, or profile requirements are not satisfied.
| `platform reconcile` | Supported | Canonical reconcile definition: a mutating, non-destructive, live-consent-gated workflow that inspects configured managed LXC-native node state and applies only the node lifecycle changes required to converge that state. Existing matching nodes report a verified no-op. Missing or stopped managed nodes may be created or started only after live consent and provider readiness pass. Missing consent, blocked provider selection, unmanaged mismatches, or unsafe evidence return blocked before mutation. CLI JSON reports mutation result as `no_op`, `converged`, `blocked`, `failed_to_apply`, or `failed_to_verify`, and reports verification as `verified`, `blocked`, or the terminal failure state.
| `platform expose` | Supported | Mutating workflow name is supported and live-consent gated. The default `lxc_native` path ensures profile-level LXC proxy devices in the manager-specific `docker-swarm-manager` profile for the setup-manifest published ports, connecting host-side listeners to `127.0.0.1:<port>` inside the manager so Swarm ingress owns service routing. It does not create direct instance-level `tsw-proxy-*` devices.
| `platform repair-lxc-proxy-drift` | Supported | Mutating workflow name is supported and live-consent gated. It is an explicit repair path for stale direct instance-level `tsw-proxy-*` devices on the manager. It removes only project-named direct proxy devices whose equivalent manager-profile device is already verified; unrelated, unknown, or non-equivalent devices are refused.
| `platform reset` and `platform destroy` | Supported | Confirmation-gated destructive workflows are wired for configured managed LXC-native nodes through LXD or Incus. They verify exact configured node names, Tiny Swarm World ownership markers, backend/profile/image evidence, and absence after deletion. Already absent managed nodes are successful reset/destroy evidence. Missing or mismatched ownership evidence blocks before mutation. Broad host cleanup such as deleting every LXD or Incus container is not a supported operation.
| `deployment bootstrap` | Supported | Workflow name is supported and live-consent gated. It starts Portainer, initializes admin access, ensures the Portainer local Docker endpoint exists, and prepares Nexus before Portainer-managed application stack apply runs.
| `artifacts prepare`, `artifacts verify`, `deployment apply`, `deployment verify` | Supported | Workflow names are supported and live-consent gated. The default `lxc_native` path uses configured artifact, image-publication, stack deployment, external-input, and readiness contracts, and still reports blocked or failed phase evidence when a live prerequisite is not observable.
| `PYTHONPATH=src python3 -m tiny_swarm_world setup run` | Supported | Safe refusal/probe command. It refuses missing live consent before setup service construction.
| `PYTHONPATH=src python3 -m tiny_swarm_world setup run --live` | Supported | Canonical live autonomous setup orchestrator. It uses `lxc_native` by default, is mutating and non-destructive, prompts for operator confirmation, selects the `service-access` profile by default, and preserves fail-closed phase states across preflight, platform, artifacts, deployment, and final verification.
| `./install.sh` | Supported | Local live fresh-install wrapper. It prompts for the exact destructive reset confirmation, or accepts the explicit `--confirm-reset` wrapper flag for deliberate test-system automation. It still runs `platform reset --live --confirm RESET_TINY_SWARM_PLATFORM`, stops if reset is blocked or failed, and only then runs the full guided setup profile. It defaults to `service-access`, passes the selected profile to reset and setup, and includes the image-packaged service-access dashboard `index.html`. The destructive scope is limited to the managed Tiny Swarm World local system; unrelated provider resources are not a supported cleanup target. Full runnable success still requires successful phase evidence from an explicit live run.
|===

== Credential And Host-Specific Data Rules

Some transitional files retain local compatibility defaults for
administrator users, passwords, registry URLs, and host addresses. They are not
production secrets and must not be copied into new product configuration.

Rules for future slices:

* prefer environment variables, CLI flags, or typed configuration ports for
  operator-supplied values;
* do not print tokens, passwords, or raw API credentials in scripts or logs;
* do not commit host-specific IP addresses, gateways, usernames, tokens, or
  passwords to new configuration;
* keep live Portainer, Nexus, Docker, LXD, Incus, LXC container lifecycle,
  netplan, socat, and stack deployment checks out of the default
  quality gate.

== Verification Policy

Normal development verification for these surfaces is static:

[source,bash]
----
git diff --check
python3 tools/quality_gate.py arch-tests
python3 tools/quality_gate.py test
----

Do not run `incus`, `lxc`, LXD or Incus daemon/profile/storage/network
changes, Docker Swarm, compose deployments, netplan, socat,
Portainer, Nexus, Jenkins, Pulsar, SonarQube, Swagger/NGINX bootstrap,
image build, image push, or stack upload commands unless the user explicitly
requests live infrastructure execution.

== Optional Live Smoke Handoff

Optional live smoke validation is an operator action, not a development quality
gate. It may be considered only after the static and mocked quality gates pass,
and only on a disposable or recoverable local environment.

[source,bash]
----
PYTHONPATH=src python3 -m tiny_swarm_world setup run --live
----

When prompted, answer `y` only after confirming that LXD/Incus/LXC provider
state, Docker Swarm, networking, Portainer, Nexus,
Jenkins, Pulsar, SonarQube, Swagger/NGINX, image, and stack changes are
intentional.

Pulsar live validation must include the Pulsar Manager UI route and the
token-authenticated Admin API check: the unauthenticated `/admin/v2/clusters`
request must be rejected and the same request with the configured
`TSW_PULSAR_ADMIN_TOKEN` bearer token must return the standalone cluster list.
Evidence must record only pass/fail status and redacted failure reasons, never
the token or the Pulsar Manager password.

Live smoke evidence must stay in ignored local state such as
`.tiny-swarm-world/`, must be redacted, and must not be committed. A
resource-gated, blocked, refused, failed-to-apply, or failed-to-verify result
is not a full runnable pass.
