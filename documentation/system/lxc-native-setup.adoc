= LXC Native Setup Through LXD Or Incus

Tiny Swarm World uses `lxc_native` as the default node provider. The provider
is managed through LXD or Incus, not through raw low-level LXC lifecycle
commands. Multipass is not a supported node-provider path.

This page documents host preparation and safe checks. It is not live proof
that a particular Windows/WSL2, LXD, Incus, Docker-in-container, or Docker
Swarm-in-container installation works.

== WSL2 Host Baseline

For WSL, use WSL2. WSL1 is not a supported provider target. The WSL
distribution must expose the Linux features needed by the selected provider,
including systemd for daemon-managed provider behavior, cgroup support, and
user namespace support.

Safe non-mutating checks from the Ubuntu WSL shell:

[source,bash]
----
systemctl is-system-running || true
test -f /sys/fs/cgroup/cgroup.controllers
test -f /proc/self/uid_map
cat /proc/sys/kernel/unprivileged_userns_clone
----

The project preflight checks these capabilities before provider mutation. A
failed WSL2 capability check blocks the default provider path; it does not
silently fall back to another provider.

== LXD Installation

Ubuntu documents the LXD snap as the recommended install path where Snap is
available. Install and grant the current user access to the LXD daemon:

[source,bash]
----
sudo snap install lxd
getent group lxd | grep -qwF "$USER" || sudo usermod -aG lxd "$USER"
newgrp lxd
----

Initialize LXD before creating instances:

[source,bash]
----
lxd init
----

Use `sudo lxd init` only when the local host policy requires root instead of
daemon access through the `lxd` group.

For a minimal local setup, use `lxd init --minimal` only when that option is
available on the installed LXD version and matches the operator's local
storage/network choice.

Verify daemon access from the same shell that runs Tiny Swarm World:

[source,bash]
----
lxc version
lxc info
----

== Incus Installation

Incus is the other supported managed LXC backend. Install it by following the
official Incus installation guide for the host distribution, then ensure the
operator can access the local daemon. Incus local administration commonly uses
the `incus-admin` group.

Verify daemon access from the same shell that runs Tiny Swarm World:

[source,bash]
----
incus version
incus info
----

For `lxc_native`, backend selection first honors an explicit
`--lxc-backend` override, then `backend_selection.preferred_backend`, then the
ordered `backend_selection.candidates` list in
`infra/config/node-providers/provider_config.yaml`. Operators may still select
a managed LXC backend explicitly:

[source,bash]
----
PYTHONPATH=src python3 -m tiny_swarm_world --lxc-backend lxd --preflight
PYTHONPATH=src python3 -m tiny_swarm_world --lxc-backend incus --preflight
----

== Project Provider Checks

The committed provider configuration lives at:

* `infra/config/node-providers/provider_config.yaml`

It declares:

* `default_provider: lxc_native`;
* ordered candidate backends `incus` and `lxd`;
* node names `swarm-manager`, `swarm-worker-1`, and `swarm-worker-2`;
* the `docker-swarm` profile requirements.

Run the non-mutating project preflight before a live run:

[source,bash]
----
PYTHONPATH=src python3 -m tiny_swarm_world --preflight
PYTHONPATH=src python3 -m tiny_swarm_world --lxc-backend lxd --preflight
PYTHONPATH=src python3 -m tiny_swarm_world --lxc-backend incus --preflight
----

The default live setup command is:

[source,bash]
----
PYTHONPATH=src python3 -m tiny_swarm_world setup run --live
----

The command still requires the live confirmation prompt. It blocks rather than
claiming success when provider readiness, profile requirements, artifacts,
deployment, or service readiness are incomplete.

== Platform Init Behavior

With accepted live consent, the default `lxc_native` Platform init path now
uses the selected managed backend to:

* ensure `swarm-manager`, `swarm-worker-1`, and `swarm-worker-2` exist;
* inspect Docker Engine state inside each managed LXC node;
* install Docker Engine inside a node only when the observed state is not
  already ready and the setup remains live-consented;
* verify Docker Engine after setup;
* initialize or detect the Swarm manager inside `swarm-manager`;
* join or verify workers from inside `swarm-worker-1` and `swarm-worker-2`.

The implementation is test-backed with fake command runners and composition
tests. It is not, by itself, live proof that a specific WSL2/LXD, WSL2/Incus,
native-Linux/LXD, or native-Linux/Incus target has succeeded. Store any
approved live-smoke result separately under the ignored local evidence tree.

Default `platform reconcile` is currently a verified no-op boundary for
`lxc_native` after Platform init. Artifact publication, service deployment,
and service readiness remain separate provider-native work and still block
until their contracts are implemented and verified.

Default `platform expose` configures host reachability for published Swarm
service ports by ensuring profile-level LXC proxy devices in the
manager-specific `docker-swarm-manager` profile. The manager container receives
that profile in addition to the common Docker Swarm profile. The proxy devices
listen on the configured host-side address and connect to
`127.0.0.1:<published-port>` inside the manager container. The Swarm ingress
mesh remains responsible for routing from the manager to the service tasks; the
workflow does not proxy individual service containers or worker nodes. Direct
instance-level `tsw-proxy-*` devices are drift and are repaired only through
the explicit `platform repair-lxc-proxy-drift` workflow after equivalent
manager-profile state is verified.

== Docker-In-Container Profile Notes

Docker Swarm remains the runtime target. Running Docker inside managed LXC
containers requires explicit profile support. The committed `docker-swarm`
profile contract requires nesting and syscall interception, keeps privileged
mode disabled by default, and forbids host network and host mounts by default.

Tiny Swarm World does not silently create or repair provider profiles during
normal quality gates. Profile creation or mutation is a live infrastructure
operation and needs explicit operator consent.

Common failure modes to check manually before rerunning live setup:

* backend readiness failure: verify the selected or ordered LXD/Incus backend,
  or select `--lxc-backend lxd` or `--lxc-backend incus` when an explicit
  override is required;
* daemon access denied: repair local LXD/Incus group or socket access outside
  Tiny Swarm World, then reopen the shell;
* profile or nesting issue: inspect the provider profile and confirm nesting
  plus syscall interception are available;
* Docker repository access failure inside a container: verify DNS and outbound
  package repository access from an operator-created disposable container.

== Manual Smoke Check

The following commands are optional operator checks and create a temporary
container. They are not part of the repository quality gate.

LXD:

[source,bash]
----
lxc launch ubuntu:24.04 test-lxc
lxc list
lxc exec test-lxc -- bash -lc "cat /etc/os-release && getent hosts archive.ubuntu.com"
lxc delete test-lxc --force
----

Incus:

[source,bash]
----
incus launch images:ubuntu/24.04 test-lxc
incus list
incus exec test-lxc -- bash -lc "cat /etc/os-release && getent hosts archive.ubuntu.com"
incus delete test-lxc --force
----

Passing this smoke check proves only the local provider/container/DNS path for
that host. It is not proof that Docker Swarm, Portainer, Nexus, Jenkins,
RabbitMQ, SonarQube, Swagger/NGINX, Vaultwarden, image publication, or stack
deployment is complete.

== Optional Live Setup Smoke Evidence

This sequence is not part of repository quality gates. Run it only after an
operator explicitly approves live infrastructure mutation for the current
target.

[source,bash]
----
export TSW_LIVE_INFRASTRUCTURE_CONSENT=I_UNDERSTAND_THIS_CHANGES_LOCAL_INFRASTRUCTURE
PYTHONPATH=src python3 -m tiny_swarm_world --preflight
PYTHONPATH=src python3 -m tiny_swarm_world setup run --live
----

When prompted, answer `y` only for an intentional local run. Classify the
target before interpreting the result:

* native Linux with LXD;
* native Linux with Incus;
* WSL2 with LXD;
* WSL2 with Incus.

Evidence checklist:

* store output under `.tiny-swarm-world/evidence/live-installation/<run-id>/`;
* keep the run target classification with the evidence;
* keep setup phase results separate from repository quality-gate output;
* redact secrets, worker join tokens, local IP addresses, and host-specific
  usernames before sharing evidence;
* do not generalize one target's result to another target class;
* do not report service-stack success unless deployment and service readiness
  phases are also verified for that run.
