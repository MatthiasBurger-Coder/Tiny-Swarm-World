= Networking

Tiny Swarm World uses Docker Swarm networking on provider nodes. The default
provider nodes are managed LXC containers through Incus.

The supported host runtime profiles are `native-linux` and `wsl2-nat`.
Treat WSL2 as its own runtime profile, not as normal native Linux. Local access
can depend on WSL NAT state, Windows portproxy rules, published Docker ports,
or provider bridge configuration. Do not hard-code addresses from examples.

Run the read-only network diagnosis before applying repairs:

[source,bash]
----
./tsw doctor network
----

== Inspect Provider Node Addresses

For Incus:

[source,bash]
----
incus list swarm-manager --format json
----

Use current provider output only for manual diagnosis. Do not commit
host-specific IP addresses, gateway values, or forwarded port state.

== Provider Network Changes

Incus network, storage, and profile changes are live infrastructure
mutations. They are not part of the default quality gate and must be performed
only when the operator intentionally approves provider changes.

Netplan helper templates for legacy VM networking have been removed from the
supported product path. Do not add host-specific netplan state to committed
configuration.

== LXC Proxy Profile Model

Published service ports are exposed through profile-level proxy devices in the
manager-specific `docker-swarm-manager` profile. The manager node receives
that profile in addition to the common Docker Swarm profile. Worker nodes keep
only worker-safe profiles and must not receive manager proxy devices.

Under `wsl2-nat`, published service ports must not depend on Docker Swarm
`ingress`/VIP routing. WSL2 kernels can lack usable `ip_vs` module support
inside managed LXC nodes, which leaves Docker's Swarm routing mesh unable to
program the published-port load balancer. Externally published Tiny Swarm
World services therefore use `mode: host` on manager-constrained services so
the LXC proxy and Windows portproxy layers have a deterministic target.

Direct instance-level `tsw-proxy-*` devices on `swarm-manager` are classified
as drift. Normal `platform init`, `platform reconcile`, `platform expose`,
`platform reset`, `install.sh`, and `setup run` do not silently remove them.
After live consent, the explicit repair command may remove a stale direct
project proxy device only when the equivalent manager-profile device is
already verified:

[source,bash]
----
PYTHONPATH=src python3 -m tiny_swarm_world platform repair-lxc-proxy-drift --live
----

Repair evidence records summary counts and device names. It does not record
raw provider command output.

== WSL2 NAT Portproxy

For WSL2, Tiny Swarm World uses controlled Windows localhost forwarding:

[source,text]
----
Windows Browser
-> http://127.0.0.1:<service-port>
-> Windows portproxy
-> current WSL eth0 IPv4
-> Incus/LXC/Docker/Services
----

Windows portproxy and firewall changes are not made by `install.sh` and are
not made implicitly from Linux. Diagnose them from elevated PowerShell:

[source,powershell]
----
powershell.exe -ExecutionPolicy Bypass -File .\tools\windows\doctor-portproxy.ps1
----

Repair them only through the explicit elevated script:

[source,powershell]
----
powershell.exe -ExecutionPolicy Bypass -File .\tools\windows\repair-wsl-portproxy.ps1
----

Both scripts read `infra/config/ports.yaml`; do not maintain a separate
Windows port list.

== Targeted Network Repair

`install.sh` must not change WSL mode, Incus runtime files, iptables/nftables,
Windows portproxy, or Windows Firewall. Use the explicit repair command and
review the dry-run first:

[source,bash]
----
./tsw network repair --runtime wsl2-nat
./tsw network repair --runtime wsl2-nat --apply
./tsw network repair --linux-forwarding --apply
./tsw network repair --incus --apply
----

The Linux forwarding repair is scoped to the Incus bridge and persists through
`tsw-incus-forwarding.service`.

== Verification

Use these checks before debugging application services:

[source,bash]
----
docker info
docker node ls
docker service ls
curl -k -s https://portainer.tsw.local
----

`docker node ls` and `docker service ls` require a Docker Swarm manager
context. Endpoint configuration and localhost routes are not service-health
proof unless observed-state or live smoke evidence exists for that run.
