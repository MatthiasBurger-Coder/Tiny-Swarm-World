= Installation

Tiny Swarm World is developed and operated from a Linux or WSL shell.
Host-specific Windows commands are legacy notes and are not the supported
default path.

== Prerequisites

Install these tools before running project automation:

* Python 3.12
* Git
* Incus or LXD for the default `lxc_native` node provider
* WSL2 with systemd, cgroup, and user-namespace support when running under WSL
* Docker CLI access from the shell that will run the scripts
* `socat`, when WSL port forwarding is needed

Optional tools for service preparation:

* `curl`
* `jq`

== Python Environment

From the repository root:

[source,bash]
----
python3 -m venv .venv
source .venv/bin/activate
python3 -m pip install --upgrade pip
python3 -m pip install -r requirements.txt
python3 -m pip install ruff mypy import-linter types-requests
----

Use the active virtual environment or the system `python3` interpreter for the
project quality gate.

== Verify The Development Setup

Run the full quality gate from the repository root:

[source,bash]
----
python3 tools/quality_gate.py quality
----

If a virtual environment is active, its interpreter can be used instead:

[source,bash]
----
.venv/bin/python tools/quality_gate.py quality
----

The quality gate sets `PYTHONPATH=src` automatically.

== LXC Native Provider Baseline

The default provider is `lxc_native`. Prepare either Incus or LXD before a
live setup run.

For LXD on Ubuntu where Snap is available:

[source,bash]
----
sudo snap install lxd
getent group lxd | grep -qwF "$USER" || sudo usermod -aG lxd "$USER"
newgrp lxd
lxd init
lxc version
lxc info
----

For Incus, follow the official installation instructions for the host
distribution, then verify access from the same shell:

[source,bash]
----
incus version
incus info
----

Run the project preflight before live mutation:

[source,bash]
----
PYTHONPATH=src python3 -m tiny_swarm_world --preflight
PYTHONPATH=src python3 -m tiny_swarm_world --lxc-backend lxd --preflight
PYTHONPATH=src python3 -m tiny_swarm_world --lxc-backend incus --preflight
----

See `documentation/system/lxc-native-setup.adoc` for the full WSL2 and
LXD/Incus setup notes and official references.

== Repository Runtime Model

Python source code lives under `src/tiny_swarm_world` and uses package imports
such as `tiny_swarm_world.application...`,
`tiny_swarm_world.domain...`, and `tiny_swarm_world.infrastructure...`.
Manual Python commands therefore need `PYTHONPATH=src` unless they are run
through `tools/quality_gate.py`.

Example:

[source,bash]
----
PYTHONPATH=src python3 -m tiny_swarm_world
----

List supported workflow-level commands without building application services:

[source,bash]
----
PYTHONPATH=src python3 -m tiny_swarm_world --list-workflows
----

== IntelliJ Execution Context

IntelliJ is supported as a Linux/WSL shell surface for this project. Configure
the run configuration to use the WSL or Linux Python interpreter, repository
root working directory, and the same environment that passes the shell
preflight checks. The command remains:

[source,bash]
----
PYTHONPATH=src python3 -m tiny_swarm_world setup run --live
----

The live confirmation prompt requires stdin. Run the command from an IntelliJ
terminal or a run configuration that can answer the prompt. This does not add
Windows-native execution support.

The safe setup probe is:

[source,bash]
----
PYTHONPATH=src python3 -m tiny_swarm_world setup run
----

Without the complete live-consent contract, `setup run` refuses before setup
services are constructed and prints `REFUSED_LIVE_CONSENT_MISSING`. This is
the expected safe default.

The canonical live operator command uses `lxc_native` by default:

[source,bash]
----
PYTHONPATH=src python3 -m tiny_swarm_world setup run --live
----

For repeatable Linux or WSL operator runs with local evidence capture, use:

[source,bash]
----
./install.sh
----

For deliberate test-system automation, the wrapper confirmation prompt can be
replaced by an explicit flag:

[source,bash]
----
./install.sh --confirm-reset
----

The wrapper writes run context, reset logs, setup logs and exit codes below
`.tiny-swarm-world/evidence/installation-tests/wsl2/<UTC timestamp>/`,
including `reset-run.log`, `reset-run.exit`, `setup-run.log`, and
`setup-run.exit`.
It loads or generates local `TSW_*` secrets in
`.tiny-swarm-world/local/live-installation.env` without printing secret
values.

`install.sh` is a fresh-install entry point. It first calls the governed
destructive reset workflow:

[source,bash]
----
PYTHONPATH=src python3 -m tiny_swarm_world platform reset --live --confirm RESET_TINY_SWARM_PLATFORM
----

The wrapper passes the selected service profile to reset and setup. The
default is `service-access`, so the effective reset command is:

[source,bash]
----
PYTHONPATH=src python3 -m tiny_swarm_world platform reset --live --confirm RESET_TINY_SWARM_PLATFORM --service-profile service-access
----

Only after reset succeeds does it call the canonical live setup workflow:

[source,bash]
----
PYTHONPATH=src python3 -m tiny_swarm_world setup run --live --service-profile service-access
----

If reset is blocked or fails, setup is not started and the script exits with
the reset exit code. The reset scope is intentionally narrow: it may discard
only the managed Tiny Swarm World local system, such as configured LXC-native
provider nodes or marker-verified project-owned instances. It must not delete
all host LXD/Incus containers or unrelated provider resources.

== Live Infrastructure Warning

The canonical setup workflow and some legacy helper areas can create or delete
provider nodes or virtual machines, change networking, install Docker, or
deploy stacks. Run live automation only when you intend to change local
infrastructure state. Former direct preparation helpers
and host-side compose
orchestration scripts have been retired.
The canonical classification for these live-operation surfaces is
`documentation/system/live-operation-surfaces.adoc`.

Mutating workflow-level commands such as `platform init`, `platform reconcile`,
`platform expose`, `platform repair-lxc-proxy-drift`, `artifacts prepare`,
`deployment apply`, and `setup run` require:

* `--live`;
* answering `y` at the short live-infrastructure confirmation prompt.

`platform reset` and `platform destroy` additionally require their exact
confirmation phrases. `install.sh` asks for
`RESET_TINY_SWARM_PLATFORM` before it runs the reset prelude unless the
operator supplies `--confirm-reset`. The flag confirms only the wrapper prompt;
the reset prelude still calls `platform reset --confirm
RESET_TINY_SWARM_PLATFORM`.

== LXC Proxy Drift Repair

`platform expose` configures published setup-manifest ports as profile-level
proxy devices in the manager-specific `docker-swarm-manager` profile. The
`swarm-manager` node receives that profile in addition to the common Docker
Swarm profile. Worker nodes do not receive the manager proxy profile.

Direct instance-level `tsw-proxy-*` devices on `swarm-manager` are not expected
normal state. They are treated as `unsafe_instance_devices` drift so normal
install, reset, reinstall, init, reconcile, expose and setup flows stop rather
than silently deleting local provider state.

When the manager-profile proxy devices are already present and verified, a
stale direct project proxy can be repaired explicitly:

[source,bash]
----
PYTHONPATH=src python3 -m tiny_swarm_world platform repair-lxc-proxy-drift --live
----

Answer `y` only when removing stale direct `tsw-proxy-*` devices from the
manager is intentional. The repair refuses unrelated devices, unknown project
proxy names, and direct project proxy devices whose equivalent
manager-profile device is missing or drifted. It reports summary counts for
removed, refused, lookup-failed and removal-failed devices without printing raw
provider command output.

Optional live smoke validation is separate from the default quality gate. Run
it only on a disposable or recoverable local environment after reviewing the
live-operation surface catalog:

[source,bash]
----
PYTHONPATH=src python3 -m tiny_swarm_world setup run --live
----

When prompted, answer `y` only if changing the local LXD/Incus/LXC provider
state, Docker Swarm, networking, Portainer, Nexus,
Jenkins, RabbitMQ, SonarQube, and Swagger/NGINX environment is intentional.

== Service Access Management Prerequisites

The full guided setup includes the service-access management stack with a
static landing page, Vaultwarden and service-access NGINX assets. The setup
CLI selects this profile by default. Operators who intentionally want only the
older base service set can pass `--service-profile default`.

The Swarm target publishes port `80` for the service-access landing page,
port `443` for the Vaultwarden HTTPS route, and port `8086` for the local HTTP
compatibility route. WSL `localhost` forwarding is a
separate host-network concern. Endpoint defaults are configuration hints, not
service-health evidence. If localhost forwarding is stale, use the current
Swarm node IP only for manual diagnosis until the forwarding issue is repaired.
The Vaultwarden administrator token is provided through an external Swarm
secret. The guided installer stores the secret value as an Argon2 PHC string in
`TSW_VAULTWARDEN_ADMIN_TOKEN`; when it sees a local plaintext token, it hashes
that value and records the original admin login token as
`TSW_VAULTWARDEN_ADMIN_LOGIN_TOKEN` in the local `0600` environment file. It
prefers the local `argon2` command and falls back to
`docker run --rm -i vaultwarden/server:latest /vaultwarden hash --preset owasp`
when `argon2` is not installed. The fallback image can be overridden with
`TSW_VAULTWARDEN_HASH_IMAGE`. By default the compose stack references
`tsw_vaultwarden_admin_token`; operators may override only the secret name through
`TSW_VAULTWARDEN_ADMIN_TOKEN_SECRET`. Document only the secret name, never the
token value.

The local `service-access` default allows initial Vaultwarden account signup.
Operators can disable that behavior with:

[source,bash]
----
export TSW_VAULTWARDEN_SIGNUPS_ALLOWED=false
----

During `deployment apply`, the setup checks that the configured external
Swarm input is observable before uploading the service-access stack. A missing
input returns a blocked deployment result with redacted evidence instead of
attempting the Portainer stack create/update.

The service-access dashboard is a static management table. It lists service
links, users and Vaultwarden item references; password values are visible only
through Vaultwarden's authenticated UI. The dashboard source is
`infra/compose/service-access/dashboard/index.html`, and the dashboard
Dockerfile packages that file into the default `service-access` setup path.
