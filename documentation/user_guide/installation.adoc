= Installation

Tiny Swarm World is developed and operated from a Linux or WSL shell.
Windows-native product execution is not supported. The elevated Windows bridge
prework documented for default WSL2 browser exposure is a supported host setup
step; project automation itself continues to run from Linux or WSL.

== Prerequisites

Install these tools before running project automation:

* Python 3.12
* Git
* Incus for the default `lxc_native` node provider
* WSL2 with systemd, cgroup, and user-namespace support when running under WSL
* `socat`, when WSL port forwarding is needed

Optional tools for service preparation:

* `curl`
* `jq`

The default `service-access` profile pulls public base images, including the
Infisical stack image, through the managed node Docker daemon. Repeated
fresh-install runs can hit Docker Hub rate limits. Use authenticated pulls on
the managed node path or provide a reviewed, operator-managed external cache
through `TSW_LXC_DOCKER_REGISTRY_MIRROR`. The former repository helper that
rewrote the host Docker daemon configuration was removed because it bypassed
the governed Python workflow.

The managed Swarm Nexus stack still prepares its own Docker proxy repository
for repository service behavior. Its defaults are:

[source,bash]
----
TSW_NEXUS_DOCKER_HUB_PROXY_REPOSITORY=docker-hub-proxy
TSW_NEXUS_DOCKER_HUB_PROXY_PORT=5001
TSW_NEXUS_DOCKER_HUB_PROXY_REMOTE_URL=https://registry-1.docker.io
----

If Docker Hub already blocks a required image and no local cache is available,
point the image variables at an approved reachable cache before running
`install.sh`:

[source,bash]
----
export TSW_NEXUS_IMAGE=registry.local:5000/sonatype/nexus3:3.75.1
export TSW_INFISICAL_IMAGE=registry.local:5000/infisical:v0.159.1
export TSW_INFISICAL_POSTGRES_IMAGE=registry.local:5000/postgres:14.23-alpine3.23
export TSW_INFISICAL_REDIS_IMAGE=registry.local:5000/redis:7.4.9-alpine3.21
----

== Python Environment

From the repository root:

[source,bash]
----
python3 -m venv .venv
source .venv/bin/activate
python3 -m pip install --upgrade pip
python3 -m pip install --require-hashes -r requirements.lock
python3 -m pip install --no-deps -e .
python3 -m pip install -r requirements-dev.txt
----

`requirements.lock` is the audited Python 3.12 runtime set used by CI. The
editable project install deliberately uses `--no-deps` so pip does not resolve
a second runtime dependency set.

Use the active virtual environment or the system `python3` interpreter for the
project quality gate.

== Verify The Development Setup

python3 tools/quality_gate.py quality
----

If a virtual environment is active, its interpreter can be used instead:

[source,bash]
----

----

== LXC Native Provider Baseline

The default provider is `lxc_native`. Prepare Incus before a live setup run.

Backend selection first honors an explicit `--lxc-backend incus` override,
then `backend_selection.preferred_backend` in
`infra/config/node-providers/provider_config.yaml`.

Backend resource names are configured under
`provider_resource_resolution.backends.<backend>`. The committed defaults map
Incus `control` traffic to `incusbr0` with `default` as the storage pool.
Override the Incus section before live setup when the local network bridge or
storage pool uses a different name.

For Incus, follow the official installation instructions for the host
distribution, then verify access from the same shell:

[source,bash]
----
incus version
incus info
----

Run the project preflight before live mutation:

[source,bash]
----
PYTHONPATH=src python3 -m tiny_swarm_world --preflight
PYTHONPATH=src python3 -m tiny_swarm_world --lxc-backend incus --preflight
----

See `documentation/system/lxc-native-setup.adoc` for the full WSL2 and
Incus setup notes and official references.

== Runtime Environment Green Path

Use this checklist when preparing a new developer workstation for the default
local runtime environment. Run project commands from a Linux or WSL shell in
the repository root. The browser can run on the host operating system, but the
automation commands remain Linux/WSL commands.

=== 1. Verify The Host Shell

Confirm that the shell is Linux or WSL2 and that the repository is reachable
through the Linux path:

[source,bash]
----
pwd
uname -a
python3 --version
git status --short --branch
----

When using WSL, verify that systemd is available:

[source,bash]
----
systemctl is-system-running || true
cat /proc/sys/kernel/unprivileged_userns_clone 2>/dev/null || true
----

`unprivileged_userns_clone` should be `1` when the file exists. If group
membership was changed for Incus access, close and reopen the shell before
continuing.

=== 2. Prepare Python Dependencies

Create the local development environment once per checkout:

[source,bash]
----
python3 -m venv .venv
source .venv/bin/activate
python3 -m pip install --upgrade pip
python3 -m pip install --require-hashes -r requirements.lock
python3 -m pip install --no-deps -e .
python3 -m pip install -r requirements-dev.txt
----

Then run the local quality gate before live infrastructure work:

[source,bash]
----
.venv/bin/python tools/quality_gate.py quality
----

=== 3. Verify Incus Baseline

The default provider is Incus-backed `lxc_native`. Verify the daemon and create
one temporary container before running Tiny Swarm World automation:

[source,bash]
----
incus version
incus info
incus launch images:ubuntu/24.04 test-incus
incus exec test-incus -- cat /etc/os-release
incus delete test-incus --force
----

The normal project setup creates and owns the managed nodes. Do not manually
create `swarm-manager`, `swarm-worker-1`, or `swarm-worker-2` unless a
documented recovery step tells you to do so.

=== 4. Prepare Kernel And Bridge Networking

Enable bridge netfilter and IPv4 forwarding on hosts that expose these
controls:

[source,bash]
----
sudo modprobe br_netfilter 2>/dev/null || true
sudo sysctl -w net.bridge.bridge-nf-call-iptables=1
sudo sysctl -w net.bridge.bridge-nf-call-ip6tables=1
sudo sysctl -w net.ipv4.ip_forward=1
----

Persist the settings when the host supports them:

[source,bash]
----
sudo tee /etc/sysctl.d/99-tiny-swarm-world.conf >/dev/null <<'EOF'
net.bridge.bridge-nf-call-iptables=1
net.bridge.bridge-nf-call-ip6tables=1
net.ipv4.ip_forward=1
EOF
sudo sysctl --system
----

Verify the effective values:

[source,bash]
----
sysctl net.ipv4.ip_forward
sysctl net.bridge.bridge-nf-call-iptables
sysctl net.bridge.bridge-nf-call-ip6tables
----

Docker Swarm overlay networking uses VXLAN. On WSL, `lsmod` may not show a
separate `vxlan` module even when VXLAN links can be created. Use a temporary
link test instead:

[source,bash]
----
DEFAULT_IFACE="$(ip route show default | awk '{print $5; exit}')"
sudo ip link add vxlan-test type vxlan id 42 dev "$DEFAULT_IFACE" dstport 4789
echo "vxlan_create=$?"
ip link show vxlan-test
sudo ip link delete vxlan-test
----

`vxlan_create=0` confirms that the kernel path is usable for the local Swarm
overlay test. Remove the test link immediately as shown above.

On native Linux hosts with a restrictive firewall, allow bridge-internal
traffic on the Incus control bridge before rerunning setup:

[source,bash]
----
sudo iptables -I FORWARD 1 -i <backend-bridge> -o <backend-bridge> -j ACCEPT
----

Replace `<backend-bridge>` with the bridge resolved from
`provider_resource_resolution.backends.<backend>.network_mappings.control`.

=== 5. Configure Local Secrets Without Printing Values

Local runtime secret files are ignored by Git and must stay local. Start from
the tracked placeholder file:

[source,bash]
----
mkdir -p .tiny-swarm-world/local
cp .env.example .tiny-swarm-world/local/live-installation.env
chmod 600 .tiny-swarm-world/local/live-installation.env
----

Edit `.tiny-swarm-world/local/live-installation.env` and replace placeholder
values. The installer may also write generated bootstrap values to the same
local secret file and recovery values to
`.tiny-swarm/secrets/generated.local.env`. Do not paste values from either
file into logs, issues, chat, commits, or documentation.

Important login-related keys include:

* `TSW_PORTAINER_USERNAME`
* `TSW_PORTAINER_ADMIN_PASSWORD`
* `TSW_PULSAR_ADMIN_TOKEN`
* `TSW_PULSAR_MANAGER_ADMIN_PASSWORD`

The setup and service-access dashboard show secret item references, not secret
values. Use the local secret file or the configured secret-management surface
to retrieve values when logging into a service.

=== 6. Configure Local Route Names

Traefik owns local HTTP and HTTPS ingress on ports `80` and `443`. Hostname
resolution must map the routed names to the local machine; DNS does not route
traffic by itself. It only makes the browser send the correct host name to
Traefik.

Prometheus, Grafana, app, and API host routes are conditional. They appear in
the generated access model, dashboard, health targets, and browser matrix only
when the corresponding service is enabled by the selected configuration.

When the services must be reachable from a Windows browser, first prepare the
Windows side from elevated PowerShell. The prerequisite action is read-only;
the install action creates the managed portproxy, Firewall, hosts-file,
Windows service discovery agent, and local bridge-state contract:

[source,powershell]
----
Set-ExecutionPolicy -Scope Process Bypass
.\tools\windows\tws-wsl-bridge.ps1 -Action prerequisites
.\tools\windows\tws-wsl-bridge.ps1 -Action install
----

See `tools/windows/README.windows-wsl-bridge.md` for WSL 2, systemd, IP Helper,
automatic discovery/reconcile, repair, status, and verification instructions.
`install.sh` stops before any platform mutation when the current discovery
heartbeat or Windows bridge state is required but missing, degraded, or stale.

For WSL/Linux shell checks, add the routed names to `/etc/hosts`. If a
matching `tsw.local` line already exists, update that line instead of adding
duplicates:

[source,bash]
----
sudo tee -a /etc/hosts >/dev/null <<'EOF'
127.0.0.1 service-access.tsw.local portainer.tsw.local nexus.tsw.local jenkins.tsw.local pulsar.tsw.local pulsar-api.tsw.local sonarqube.tsw.local swagger.tsw.local infisical.tsw.local
EOF
----

When opening the services from a Windows browser against a WSL-hosted runtime,
add the same hostnames to the Windows host resolver with Administrator rights,
or configure a local DNS server that resolves `*.tsw.local` to `127.0.0.1`.
Use a wildcard DNS entry only for name resolution; Traefik still needs the
service-access stack and the published `80` and `443` ingress ports.

=== 7. Run The Fresh Install

Use the default `service-access` profile for the full local runtime. The
current LXC Swarm ingress path requires an explicit privileged-ingress opt-in
when the managed Incus profile needs privileged Swarm settings:

[source,bash]
----
export TSW_LXC_ALLOW_PRIVILEGED_SWARM_INGRESS=1
./install.sh --headless --confirm-reset --non-interactive-live-approval
----

Without `--headless`, the installer uses the interactive terminal recorder.
Without `--confirm-reset`, it asks for the exact reset phrase before the
fresh-install reset. Without `--non-interactive-live-approval`, it asks the
normal live-infrastructure confirmation prompt.

The wrapper first runs:

[source,bash]
----
PYTHONPATH=src python3 -m tiny_swarm_world platform reset --live --confirm RESET_TINY_SWARM_PLATFORM --service-profile service-access
----

Only after reset succeeds does it run:

[source,bash]
----
PYTHONPATH=src python3 -m tiny_swarm_world setup run --live --service-profile service-access
----

The run writes evidence under
`.tiny-swarm-world/evidence/installation-tests/<host-runtime>/<UTC timestamp>/`.
Use `reset-run.exit`, `setup-run.exit`, and the matching logs to diagnose a
failed or blocked run.

Before stack apply, the deployment workflow also writes the selected effective
access model to
`.tiny-swarm-world/evidence/solid-typed-evidence/routing/effective-access-model.json`.
This redacted file contains configured routing evidence and only credential
labels plus Infisical item references. Its `result: generated` value does not
prove that a route is deployed or reachable.

=== 8. Verify The Runtime

After installation, verify the managed nodes and Swarm state:

[source,bash]
----
incus list --format compact
incus exec swarm-manager -- docker node ls
incus exec swarm-manager -- docker stack ls
incus exec swarm-manager -- docker service ls
----

Then verify the routed service surfaces from the same environment where the
browser will run:

[source,bash]
----
curl -k -sS -o /dev/null -w 'service-access %{http_code}\n' https://service-access.tsw.local/
curl -k -sS -o /dev/null -w 'portainer %{http_code}\n' https://portainer.tsw.local/
curl -k -sS -o /dev/null -w 'nexus %{http_code}\n' https://nexus.tsw.local/
curl -k -sS -o /dev/null -w 'jenkins %{http_code}\n' https://jenkins.tsw.local/login
curl -k -sS -o /dev/null -w 'pulsar-manager %{http_code}\n' https://pulsar.tsw.local/
curl -k -sS -o /dev/null -w 'pulsar-api %{http_code}\n' https://pulsar-api.tsw.local/admin/v2/clusters
curl -k -sS -o /dev/null -w 'sonarqube %{http_code}\n' https://sonarqube.tsw.local/api/system/status
curl -k -sS -o /dev/null -w 'swagger %{http_code}\n' https://swagger.tsw.local/
curl -k -sS -o /dev/null -w 'infisical %{http_code}\n' https://infisical.tsw.local/
----

For authentication-protected endpoints, `401`, `403`, or a login page can be
valid reachability evidence. For the Pulsar Admin API, a request without the
bearer token should be treated as a route/authentication check, not as a
service failure. A token-authenticated check can be run without printing the
token:

[source,bash]
----
set -a
. .tiny-swarm-world/local/live-installation.env
set +a
curl -k -sS -H "Authorization: Bearer ${TSW_PULSAR_ADMIN_TOKEN}" \
  https://pulsar-api.tsw.local/admin/v2/clusters
----

The local standalone Pulsar green path returns a cluster list that includes
`standalone`.

=== 10. Re-run And Recovery Rules

The setup path is designed to be rerunnable. Prefer rerunning the same
installer command after fixing host prerequisites:

[source,bash]
----
export TSW_LXC_ALLOW_PRIVILEGED_SWARM_INGRESS=1
./install.sh --headless --confirm-reset --non-interactive-live-approval
----

If a developer intentionally needs to discard the managed local environment,
use the governed reset path rather than deleting arbitrary Incus resources:

[source,bash]
----
PYTHONPATH=src python3 -m tiny_swarm_world platform reset --live --confirm RESET_TINY_SWARM_PLATFORM --service-profile service-access
----

If direct `tsw-proxy-*` instance devices block reset or expose operations,
repair only the reviewed Tiny Swarm World proxy drift:

[source,bash]
----
PYTHONPATH=src python3 -m tiny_swarm_world platform repair-lxc-proxy-drift --live
----

Do not commit `.tiny-swarm-world/`, `.tiny-swarm/`, `.venv/`, coverage files,
or any generated local secret file.

== IntelliJ Execution Context

IntelliJ is supported as a Linux/WSL shell surface for this project. Configure
the run configuration to use the WSL or Linux Python interpreter, repository
root working directory, and the same environment that passes the shell
preflight checks. The command remains:

[source,bash]
----
tiny-swarm-world setup run --live
----

The live confirmation prompt requires stdin. Run the command from an IntelliJ
terminal or a run configuration that can answer the prompt. This does not add
Windows-native execution support.

The safe setup probe is:

[source,bash]
----
tiny-swarm-world setup run
----

Without the complete live-consent contract, `setup run` refuses before setup
services are constructed and prints `REFUSED_LIVE_CONSENT_MISSING`. This is
the expected safe default.

The canonical live operator command uses `lxc_native` by default:

[source,bash]
----
tiny-swarm-world setup run --live
----

For repeatable Linux or WSL operator runs with local evidence capture, use:

[source,bash]
----
./install.sh
----

`install.sh` is intentionally a minimal Linux/WSL bootstrapper. The
authoritative installer policy is `python3 -m tiny_swarm_world.installer`:
that Python entry point owns secret handling, host-runtime detection, evidence
layout, reset/setup sequencing, dependency bootstrap, and headless
execution. The shell wrapper only verifies the repository shape and delegates.

For deliberate test-system automation, the wrapper confirmation prompt can be
replaced by an explicit flag:

[source,bash]
----
./install.sh --confirm-reset
----

The CLI live-infrastructure prompt is separate from the reset prompt. Normal
wrapper runs keep that prompt interactive inside the terminal recorder. For
deliberate non-interactive automation, pass an explicit live-approval flag:

[source,bash]
----
./install.sh --non-interactive-live-approval
PYTHONPATH=src python3 -m tiny_swarm_world setup run --live --approve-live
----

The automation flag is an approval source; it does not remove the `--live`
requirement.

Terminal recording is optional presentation. For headless systems that cannot
or should not start the terminal recorder/TUI surface, use:

[source,bash]
----
./install.sh --headless
TSW_INSTALL_HEADLESS=1 ./install.sh
----

Headless mode executes the same governed reset and setup commands, writes the
same `reset-run.log`, `reset-run.exit`, `setup-run.log`, and `setup-run.exit`
evidence files, and preserves command exit codes.

The wrapper writes run context, reset logs, setup logs and exit codes below
`.tiny-swarm-world/evidence/installation-tests/<host-runtime>/<UTC timestamp>/`.
The stable host directory is `wsl2` for WSL2 and `native_linux` for native
Linux. The run `context.txt` records `host_runtime_type`,
`host_runtime_detection_source`, `selected_evidence_directory`,
`live_execution_mode`, `live_approval_source`, and `terminal_recording_mode`
alongside `reset-run.log`, `reset-run.exit`, `setup-run.log`, and
`setup-run.exit`.
It loads or generates local `TSW_*` secrets in
`.tiny-swarm-world/local/live-installation.env` without printing secret
values.
Required local bootstrap values are derived from
`infra/config/secrets/infisical-secrets.yaml`; installer code does not maintain
a separate required-secret list.
Use `.env.example` as the tracked placeholder template. Before live setup,
copy the needed keys into `.tiny-swarm-world/local/live-installation.env` or
export them in the process environment. Process environment values override
the local file, and preflight validates the resulting contract before setup
continues.
On Linux or WSL, if the runtime Python dependencies from `requirements.lock`
are not importable, `install.sh` creates an ignored local environment at
`.tiny-swarm-world/install-venv` and uses its Python interpreter for the reset
and setup commands. That bootstrap installs the lock with hash verification,
then installs the editable project without resolving dependencies again. WSL
keeps the existing `python3` execution path when the required imports are
already available.
Ensure the current shell can access the Incus daemon before running
`install.sh`; reopen the shell after group membership changes when required by
the host distribution.

On native Linux, Docker Swarm also requires that the managed LXC containers can
reach each other on the selected backend bridge. Use the bridge name resolved
from `provider_resource_resolution.backends.<backend>.network_mappings.control`
when applying host firewall rules. If workers can reach the configured bridge
gateway but cannot ping the manager container or connect to Swarm port `2377`,
allow bridge-internal forwarding on the host:

[source,bash]
----
sudo iptables -I FORWARD 1 -i <backend-bridge> -o <backend-bridge> -j ACCEPT
----

Verify the host firewall path before rerunning installation. Replace the
manager address with the current manager container address from
`incus list` or `incus exec swarm-manager -- hostname -I`:

[source,bash]
----
incus exec swarm-worker-1 -- ping -c 2 <manager-ip>
incus exec swarm-worker-1 -- nc -vz <manager-ip> 2377
----

If `ufw` manages the host firewall, persist the bridge route rule with:

[source,bash]
----
sudo ufw route allow in on <backend-bridge> out on <backend-bridge>
sudo ufw reload
----

On hosts that use persistent iptables rules directly, save the accepted
FORWARD rule after validating it:

[source,bash]
----
sudo apt install iptables-persistent
sudo netfilter-persistent save
----

Swarm bootstrap needs container-to-container access for `2377/tcp`, and later
Swarm traffic uses `7946/tcp`, `7946/udp`, and `4789/udp`. The bridge-internal
FORWARD rule above allows that managed backend-node traffic on the configured
bridge.

Portainer stack create and update calls can take longer than ordinary API
lookups while Docker Swarm pulls images or reconciles services. The installer
uses a longer Portainer stack operation timeout by default. On especially slow
hosts, raise it before rerunning the idempotent installation:

[source,bash]
----
export TSW_PORTAINER_STACK_REQUEST_TIMEOUT_SECONDS=300
./install.sh --confirm-reset
----

`install.sh` is a fresh-install entry point. It first calls the governed
destructive reset workflow:

[source,bash]
----
PYTHONPATH=src python3 -m tiny_swarm_world platform reset --live --confirm RESET_TINY_SWARM_PLATFORM
----

The wrapper passes the selected service profile to reset and setup. The
default is `service-access`, so the effective reset command is:

[source,bash]
----
PYTHONPATH=src python3 -m tiny_swarm_world platform reset --live --confirm RESET_TINY_SWARM_PLATFORM --service-profile service-access
----

Only after reset succeeds does it call the canonical live setup workflow:

[source,bash]
----
PYTHONPATH=src python3 -m tiny_swarm_world setup run --live --service-profile service-access
----

If reset is blocked or fails, setup is not started and the script exits with
the reset exit code. The reset scope is intentionally narrow: it may discard
only the managed Tiny Swarm World local system, such as configured LXC-native
provider nodes or marker-verified project-owned instances. It must not delete
all host Incus containers or unrelated provider resources.

== Live Infrastructure Warning

The canonical setup workflow and some legacy helper areas can create or delete
provider nodes or virtual machines, change networking, install Docker, or
deploy stacks. Run live automation only when you intend to change local
infrastructure state. Former direct preparation helpers
and host-side compose
orchestration scripts have been retired.
The canonical classification for these live-operation surfaces is
`documentation/system/live-operation-surfaces.adoc`.

Mutating workflow-level commands such as `platform init`, `platform reconcile`,
`platform expose`, `platform repair-lxc-proxy-drift`, `artifacts prepare`,
`deployment apply`, and `setup run` require:

* `--live`;
* answering `y` at the short live-infrastructure confirmation prompt.

`platform reset` and `platform destroy` additionally require their exact
confirmation phrases. `install.sh` asks for
`RESET_TINY_SWARM_PLATFORM` before it runs the reset prelude unless the
operator supplies `--confirm-reset`. The flag confirms only the wrapper prompt;
the reset prelude still calls `platform reset --confirm
RESET_TINY_SWARM_PLATFORM`.

== LXC Proxy Drift Repair

`platform expose` configures published setup-manifest ports as profile-level
proxy devices in the manager-specific `docker-swarm-manager` profile. The
`swarm-manager` node receives that profile in addition to the common Docker
Swarm profile. Worker nodes do not receive the manager proxy profile.

Direct instance-level `tsw-proxy-*` devices on `swarm-manager` are not expected
normal state. They are treated as `unsafe_instance_devices` drift so normal
install, reset, reinstall, init, reconcile, expose and setup flows stop rather
than silently deleting local provider state.

When the manager-profile proxy devices are already present and verified, a
stale direct project proxy can be repaired explicitly:

[source,bash]
----
PYTHONPATH=src python3 -m tiny_swarm_world platform repair-lxc-proxy-drift --live
----

Answer `y` only when removing stale direct `tsw-proxy-*` devices from the
manager is intentional. The repair refuses unrelated devices, unknown project
proxy names, and direct project proxy devices whose equivalent
manager-profile device is missing or drifted. It reports summary counts for
removed, refused, lookup-failed and removal-failed devices without printing raw
provider command output.

Optional live smoke validation is separate from the default quality gate. Run
it only on a disposable or recoverable local environment after reviewing the
live-operation surface catalog:

[source,bash]
----
PYTHONPATH=src python3 -m tiny_swarm_world setup run --live
----

When prompted, answer `y` only if changing the local Incus/LXC provider
state, Docker Swarm, networking, Portainer, Nexus,
Jenkins, Apache Pulsar, SonarQube, and Swagger/NGINX environment is intentional.

== Service Access Management Prerequisites

The full guided setup includes the service-access management stack with a
dashboard asset rendered from the effective service-access model and
service-access NGINX assets. The setup CLI selects this profile by default.
Operators who intentionally want only the older base service set can pass
`--service-profile default`.

The service-access setup profile exposes Traefik ingress ports `80` and `443`
through the LXC/WSL exposure workflow. The service-access NGINX compose
service publishes ports `10000` and `8086` as stack metadata parsed from

The service-access dashboard is a generated management table. It lists
service links, users and Infisical item references from the effective access
model; password values are visible only through the configured
secret-management surface. Preferred links use routed HTTPS URLs without the
diagnostic ports `10080` or `10443`. Runtime renderer output is authoritative;
the committed `infra/config/compose/service-access/dashboard/index.html` is a
default-profile fallback/image asset guarded by a renderer drift test, and the
dashboard Dockerfile packages it into the default `service-access` setup path.
