= Troubleshooting

== Quality Gate Fails: Missing Tool

If Ruff, mypy, import-linter, or the Requests type stubs are missing, install
the development tools in the active Python environment:

[source,bash]
----
python3 -m pip install ruff mypy import-linter types-requests
----

Then rerun:

[source,bash]
----
python3 tools/quality_gate.py quality
----

== Import Errors For tiny_swarm_world

Manual Python commands need the `src/` directory on `PYTHONPATH`:

[source,bash]
----
PYTHONPATH=src python3 -m unittest discover -s tests -t .
----

The quality gate configures this automatically.

== import-linter Cannot Find Configuration

The architecture lint gate expects `.importlinter` in the repository root.
Run the gate from the repository root, not from a source or test subdirectory.

== LXC Native Provider Is Not Ready

The default node provider is `lxc_native` through LXD or Incus. Check the
selected backend from the same Linux/WSL shell that will run Tiny Swarm World.

For Incus:

[source,bash]
----
incus version
incus info
----

For LXD:

[source,bash]
----
lxc version
lxc info
----

If both backends are installed and usable, select one explicitly:

[source,bash]
----
PYTHONPATH=src python3 -m tiny_swarm_world --lxc-backend incus --preflight
PYTHONPATH=src python3 -m tiny_swarm_world --lxc-backend lxd --preflight
----

In WSL, verify that the distribution is WSL2 and that systemd, cgroup, and
user namespace support are available. WSL1 is not a supported target for the
default provider.

Provider readiness checks are read-only. They do not install packages, start
or repair daemons, create profiles, change group membership, or mutate
provider networking.

== Multipass Legacy/Fallback Is Not Available

Multipass is relevant only for explicit legacy/fallback runs:

[source,bash]
----
PYTHONPATH=src python3 -m tiny_swarm_world --node-provider multipass_legacy setup run --live
----

Verify Multipass installation and driver state:

[source,bash]
----
multipass version
multipass get local.driver
multipass info
----

For legacy WSL/Linux usage, use the QEMU driver.

If an explicit legacy run reports a `RUNTIME-MULTIPASS-*` blocker, repair
Multipass access from the same WSL or Linux shell before retrying. The
readiness check is read-only; it does not start services, change drivers,
create VMs, or repair socket permissions.

== IntelliJ Run Configuration Cannot Start Live Setup

Use IntelliJ as a WSL/Linux shell launcher. The run configuration must use the
WSL or Linux interpreter, the repository root as working directory, and the
same environment as a successful terminal run.

[source,bash]
----
PYTHONPATH=src python3 -m tiny_swarm_world setup run --live
----

The live prompt needs stdin. If the IDE runner cannot answer the prompt, use
the IntelliJ terminal or an external WSL/Linux terminal.

== Docker Or Portainer Calls Fail

Check that Docker is reachable from the same shell:

[source,bash]
----
docker info
----

Check Portainer manually before running stack upload or Nexus bootstrap:

[source,bash]
----
curl -s http://localhost:9000
----

== Setup Preflight Reports Occupied Ports

`setup run --live` is intentionally re-runnable. During preflight, an occupied
port is accepted only when the local host probe recognizes the expected Tiny
Swarm World service on that port.

Expected already-running services are reported with
`source=existing_expected_service` and the port check passes. Unknown port
owners still fail preflight and must be stopped or remapped before live
execution.

If a port fails although the service should already be running, verify the
service from the same WSL shell:

[source,bash]
----
curl -s http://localhost:9000/api/status
curl -s http://localhost:8081/service/rest/v1/status
curl -s http://localhost:5000/v2/
curl -s http://localhost:8080/login
curl -s http://localhost:15672
curl -s http://localhost:9001/api/system/status
----

== Nexus Bootstrap Cannot Authenticate

Use the canonical setup workflow and provide explicit credentials through
environment variables where supported:

[source,bash]
----
TSW_PORTAINER_USERNAME=admin \
TSW_PORTAINER_PASSWORD='<portainer-password>' \
TSW_NEXUS_ADMIN_PASSWORD='<new-nexus-admin-password>' \
PYTHONPATH=src python3 -m tiny_swarm_world setup run --live
----

If Nexus is still starting, increase:

* `TSW_NEXUS_MAX_ATTEMPTS`
* `TSW_NEXUS_WAIT_SECONDS`

== Service Access Or Vaultwarden Is Not Reachable

The full guided setup selects the service-access profile by default, but the
default `lxc_native` path still blocks before provider-native image
publication, stack deployment, and Swarm service-readiness verification are
complete. Service-access dashboard and NGINX assets are committed repository
inputs until a live provider-specific deployment path verifies them.

The committed dashboard is static catalog data. It intentionally renders
service reachability as unknown until observed evidence exists. Password
values are not present in the dashboard; open Vaultwarden and use the listed
item name only after Vaultwarden itself is deployed and authenticated access is
available.

Check the non-secret prerequisites before a live run:

* the Swarm nodes expose `80` for the service-access landing page;
* the Swarm nodes expose `8086` for Vaultwarden;
* `TSW_VAULTWARDEN_ADMIN_TOKEN_SECRET` names an external Swarm secret, or the
  default Swarm secret name `tsw_vaultwarden_admin_token` exists;
* a missing external Swarm input blocks before the service-access stack is
  uploaded through Portainer;
* the service-access dashboard and NGINX images are available to the Swarm
  registry;
* the service-access stack has been deployed intentionally through a reviewed
  live path.

In WSL, `localhost` and `localhost:8086` depend on host-forwarding state
and may be out of sync even when the stack is reachable on the Swarm node IPs.
The service-access setup profile treats an unexpected local listener on those
ports as a mandatory preflight blocker. If `http://localhost` returns an
unexpected page, clear the stale local listener or redeploy the updated
service-access stack.

Endpoint configuration does not prove service health. Treat `http://localhost`
routes as operator defaults only; use deployment verification or explicit live
smoke evidence before claiming a service is reachable.

== Git Shows Generated Python Files

Generated files such as `__pycache__/`, `.ruff_cache/`, `.mypy_cache/`, and
PyCharm debugger folders are ignored by `.gitignore`. If they still appear,
verify that the current branch includes the repository `.gitignore` changes and
run:

[source,bash]
----
git status --ignored --short
----

== Git Shows .tiny-swarm-world Files

Observed inventory and verification evidence are local runtime artifacts under
`.tiny-swarm-world/`. The directory is ignored and must not be committed. If
files below that path appear as tracked changes, stop and verify whether they
were accidentally added before continuing.

== REFUSED_LIVE_CONSENT_MISSING

Mutating workflow-level commands such as `setup run`, `platform init`,
`platform reconcile`, `artifacts prepare`, and `deployment apply` require all
live-consent controls. For the canonical setup path:

[source,bash]
----
PYTHONPATH=src python3 -m tiny_swarm_world setup run --live
----

When prompted:

[source,text]
----
Live setup can change local infrastructure. Continue? [y/N]:
----

Answer `y` only when changing local infrastructure is intentional. The
workflow refuses before building application services if `--live` or the
short live confirmation is missing.

This refusal is expected for a plain `setup run` command without consent. It
does not mean the default quality gate failed.

== REFUSED_WORKFLOW_CONFIRMATION_MISSING

`platform reset` and `platform destroy` require exact confirmation:

[source,bash]
----
PYTHONPATH=src python3 -m tiny_swarm_world platform reset --confirm RESET_TINY_SWARM_PLATFORM --live
PYTHONPATH=src python3 -m tiny_swarm_world platform destroy --confirm DESTROY_TINY_SWARM_PLATFORM --live
----

After confirmation, reset and destroy currently return `blocked` until
retention and teardown semantics are implemented. Do not treat confirmation as
proof that destructive teardown is wired.

== Git Shows Line Ending Only Changes

The repository uses `.gitattributes` to normalize text files to LF. If a large
number of files appears modified after switching between Windows Git and WSL,
refresh the checkout with Git after saving any real work.

Do not commit line-ending-only changes unless the change is an intentional
normalization commit.
