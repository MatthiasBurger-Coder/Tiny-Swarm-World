= Troubleshooting

== Quality Gate Fails: Missing Tool

If Ruff, mypy, import-linter, or the Requests type stubs are missing, install
the development tools in the active Python environment:

[source,bash]
----
python3 -m pip install ruff mypy import-linter types-requests
----

Then rerun:

[source,bash]
----
python3 tools/quality_gate.py quality
----

== Import Errors For tiny_swarm_world

Manual Python commands need the `src/` directory on `PYTHONPATH`:

[source,bash]
----
PYTHONPATH=src python3 -m unittest discover -s tests -t .
----

The quality gate configures this automatically.

== import-linter Cannot Find Configuration

The architecture lint gate expects `.importlinter` in the repository root.
Run the gate from the repository root, not from a source or test subdirectory.

== LXC Native Provider Is Not Ready

The default node provider is `lxc_native` through LXD or Incus. Check the
selected backend from the same Linux/WSL shell that will run Tiny Swarm World.

For Incus:

[source,bash]
----
incus version
incus info
----

For LXD:

[source,bash]
----
lxc version
lxc info
----

If the configured backend order should not be used for this run, select one
explicitly:

[source,bash]
----
PYTHONPATH=src python3 -m tiny_swarm_world --lxc-backend incus --preflight
PYTHONPATH=src python3 -m tiny_swarm_world --lxc-backend lxd --preflight
----

In WSL, verify that the distribution is WSL2 and that systemd, cgroup, and
user namespace support are available. WSL1 is not a supported target for the
default provider.

Provider readiness checks are read-only. They do not install packages, start
or repair daemons, create profiles, change group membership, or mutate
provider networking.

== Unsupported Node Provider Selection

The supported node provider is `lxc_native`. If an old command line uses a
removed provider value, rerun with the default provider or select an LXD/Incus
backend explicitly:

[source,bash]
----
PYTHONPATH=src python3 -m tiny_swarm_world setup run --live
PYTHONPATH=src python3 -m tiny_swarm_world --lxc-backend incus setup run --live
PYTHONPATH=src python3 -m tiny_swarm_world --lxc-backend lxd setup run --live
----

== IntelliJ Run Configuration Cannot Start Live Setup

Use IntelliJ as a WSL/Linux shell launcher. The run configuration must use the
WSL or Linux interpreter, the repository root as working directory, and the
same environment as a successful terminal run.

[source,bash]
----
PYTHONPATH=src python3 -m tiny_swarm_world setup run --live
----

The live prompt needs stdin. If the IDE runner cannot answer the prompt, use
the IntelliJ terminal or an external WSL/Linux terminal.

== Docker Or Portainer Calls Fail

Check that Docker is reachable from the same shell:

[source,bash]
----
docker info
----

Check Portainer manually before running stack upload or Nexus bootstrap:

[source,bash]
----
curl -k -s https://portainer.tsw.local
----

== Portainer Admin Initialization Returns HTTP 409

`PortainerAdminInitializationRejected HTTP 409` means Portainer rejected the
administrator initialization call. In the fresh-install path, `install.sh`
first runs the managed platform reset, so a 409 after that reset is evidence
to review rather than a readiness delay.

The deployment workflow treats a rejected initialization as already achieved
only when the same configured Portainer username and password authenticate
immediately afterward. If those requested credentials do not authenticate,
the failure stays explicit because the remaining cause is usually stale or
unmanaged Portainer state, an incomplete reset, or a credential mismatch.

Check the reset evidence first:

[source,bash]
----
RUN_ID=20260604T120000Z
HOST_RUNTIME=wsl2
EVIDENCE_DIR=".tiny-swarm-world/evidence/installation-tests/$HOST_RUNTIME/$RUN_ID"
cat "$EVIDENCE_DIR/reset-run.exit"
cat "$EVIDENCE_DIR/setup-run.exit"
----

Use the concrete host runtime and run ID printed by the wrapper. Supported
host runtime directories are `wsl2` and `native_linux`; the run context file
also records `host_runtime_type` and `selected_evidence_directory`.

Then verify that the operator-provided `TSW_PORTAINER_USERNAME` and
`TSW_PORTAINER_ADMIN_PASSWORD` values are the intended values for this local
installation. Do not paste password values into issue reports, logs, or
committed documentation.

== Installation Debugger For WSL2

Use the installation debugger when `install.sh` aborts and the next question is
whether the checkout, host runtime, WSL2 systemd availability, service
definitions, or evidence logs are the problem:

[source,bash]
----
python3 tools/install_debugger.py
----

The default mode analyzes the `install.sh` bootstrap contract, repository
permissions, ownership markers, WSL2/systemd indicators, systemd unit
definitions, and the newest installation evidence directory across supported
host directories. `install.sh` must delegate to
`python3 -m tiny_swarm_world.installer`; the Python installer owns the reset,
setup, terminal recording, headless logging, and evidence policy. The debugger
also prints exact `grep`, `tail`, `systemctl`, and `journalctl` commands for
finding the abort point after a `Service` log line.

For live host diagnostics, including `systemctl is-system-running`, use:

[source,bash]
----
python3 tools/install_debugger.py --live
----

To allow narrow repository permission repairs, add `--fix-permissions`:

[source,bash]
----
python3 tools/install_debugger.py --live --fix-permissions
----

This repair mode is intentionally limited to the installer execute bit and the
local secret-file mode. Ownership mismatches are reported with a review command
instead of being changed automatically.

== Setup Preflight Reports Occupied Ports

`setup run --live` is intentionally re-runnable. During preflight, an occupied
port is accepted only when the local host probe recognizes the expected Tiny
Swarm World service on that port.

Expected already-running services are reported with
`source=existing_expected_service` and the port check passes. Unknown port
owners still fail preflight and must be stopped or remapped before live
execution.

If a port fails although the service should already be running, verify the
service from the same WSL shell:

[source,bash]
----
curl -k -s https://portainer.tsw.local/api/status
curl -k -s https://nexus.tsw.local/service/rest/v1/status
curl -s http://localhost:13500/v2/
curl -k -s https://jenkins.tsw.local/login
curl -k -s https://pulsar-api.tsw.local/admin/v2/clusters
curl -k -s https://sonarqube.tsw.local/api/system/status
----

== Nexus Bootstrap Cannot Authenticate

Use the canonical setup workflow and provide explicit credentials through
environment variables where supported:

[source,bash]
----
TSW_PORTAINER_USERNAME=admin \
TSW_PORTAINER_ADMIN_PASSWORD='<portainer-password>' \
TSW_NEXUS_ADMIN_PASSWORD='<new-nexus-admin-password>' \
PYTHONPATH=src python3 -m tiny_swarm_world setup run --live
----

If Nexus is still starting, increase:

* `TSW_NEXUS_MAX_ATTEMPTS`
* `TSW_NEXUS_WAIT_SECONDS`

== Service Access Is Not Reachable

The full guided setup selects the service-access profile by default. The
default `lxc_native` path uses configured artifact image-publication, stack
deployment, and Swarm service-readiness contracts, but each phase can still
block or fail when live prerequisites such as registry access, external Swarm
inputs, or observed service readiness are unavailable. Service-access
dashboard and NGINX assets are committed repository inputs and are packaged
into their respective images by the artifact workflow; live reachability still
requires explicit deployment evidence.

The committed dashboard is generated from the effective service-access model.
It intentionally does not claim service reachability until observed evidence
exists. Password values are not present in the dashboard; use the listed
secret item reference only through the configured secret-management surface.

Check the non-secret prerequisites before a live run:

* the service-access setup profile exposes Traefik ingress ports `80` and
  `443`;
* the service-access NGINX compose service publishes `10000` for the dashboard
  and `8086` for the Infisical compatibility route as stack metadata;
* the service-access dashboard and NGINX images are available to the Swarm
  registry;
* the service-access stack has been deployed intentionally through a reviewed
  live path.

In WSL, `localhost`, `localhost:10000`, and `localhost:8086` depend on host-forwarding state
and may be out of sync even when the stack is reachable on the Swarm node IPs.
The service-access setup profile treats an unexpected local listener on those
ports as a mandatory preflight blocker. If `http://localhost:10000` returns an
unexpected page, clear the stale local listener or redeploy the updated
service-access stack.

Preferred HTTP access uses Traefik host routes such as
`https://service-access.tsw.local`; direct localhost ports are diagnostic or
compatibility surfaces.

Endpoint configuration does not prove service health. Treat local route
defaults as operator targets only; use deployment verification or explicit live
smoke evidence before claiming a service is reachable.

== Git Shows Generated Python Files

Generated files such as `__pycache__/`, `.ruff_cache/`, `.mypy_cache/`, and
PyCharm debugger folders are ignored by `.gitignore`. If they still appear,
verify that the current branch includes the repository `.gitignore` changes and
run:

[source,bash]
----
git status --ignored --short
----

== Git Shows .tiny-swarm-world Files

Observed inventory and verification evidence are local runtime artifacts under
`.tiny-swarm-world/`. The directory is ignored and must not be committed. If
files below that path appear as tracked changes, stop and verify whether they
were accidentally added before continuing.

== REFUSED_LIVE_CONSENT_MISSING

Mutating workflow-level commands such as `setup run`, `platform init`,
`platform reconcile`, `platform expose`, `artifacts prepare`, and
`deployment apply` require all
live-consent controls. For the canonical setup path:

[source,bash]
----
PYTHONPATH=src python3 -m tiny_swarm_world setup run --live
----

When prompted:

[source,text]
----
Live setup can change local infrastructure. Continue? [y/N]:
----

Answer `y` only when changing local infrastructure is intentional. The
workflow refuses before building application services if `--live` or the
short live confirmation is missing.

This refusal is expected for a plain `setup run` command without consent. It
does not mean the default quality gate failed.

== REFUSED_WORKFLOW_CONFIRMATION_MISSING

`platform reset` and `platform destroy` require exact confirmation:

[source,bash]
----
PYTHONPATH=src python3 -m tiny_swarm_world platform reset --confirm RESET_TINY_SWARM_PLATFORM --live
PYTHONPATH=src python3 -m tiny_swarm_world platform destroy --confirm DESTROY_TINY_SWARM_PLATFORM --live
----

After confirmation, reset and destroy act only on configured managed
LXC-native nodes whose Tiny Swarm World ownership evidence is verified before
mutation. Already absent managed nodes are valid reset/destroy evidence.
Missing, mismatched, or unmanaged ownership evidence blocks before deletion.
Do not use these commands as broad LXD/Incus cleanup, and do not treat a
confirmation phrase as permission to delete unrelated containers.

If reset reports `managed_nodes_reset_blocked` with
`first_failure_classification=reset_existing_node_not_managed`, a configured
node name already exists but does not carry the expected Tiny Swarm World
ownership markers. Inspect the reset evidence for `first_failure_node` and
`first_failure_mismatch_reasons`. Resolve that named container manually before
rerunning the installer: keep it if it is unrelated, rename/remove it only if
you intentionally own that local resource, or recreate it through the guarded
setup workflow so the managed markers are written by automation.
Direct instance-level `tsw-proxy-<port>` devices are drift. Current
`platform expose` manages equivalent proxy devices in the manager-specific
profile instead of creating direct manager instance devices. If reset reports
`unsafe_instance_devices` for stale direct project proxy devices, first ensure
the manager-profile devices are present, then run the explicit repair command:

[source,bash]
----
PYTHONPATH=src python3 -m tiny_swarm_world platform repair-lxc-proxy-drift --live
----

The repair refuses devices whose equivalent manager-profile state is missing
or drifted. Any remaining `unsafe_instance_devices` evidence points to direct
runtime state that must be inspected and resolved intentionally.

== Git Shows Line Ending Only Changes

The repository uses `.gitattributes` to normalize text files to LF. If a large
number of files appears modified after switching between Windows Git and WSL,
refresh the checkout with Git after saving any real work.

Do not commit line-ending-only changes unless the change is an intentional
normalization commit.
