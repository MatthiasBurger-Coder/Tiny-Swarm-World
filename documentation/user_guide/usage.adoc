= Usage

Tiny Swarm World contains two different kinds of workflows:

* development workflows that are safe to run during normal coding
* live infrastructure workflows that change LXD/Incus/LXC provider state,
  Docker, Portainer, Nexus, networking, or service stacks

Use a Linux or WSL shell from the repository root unless a command explicitly
states otherwise.

== Development Quality Gate

Run the full gate before handing off changes:

[source,bash]
----
python3 tools/quality_gate.py quality
----

Available subcommands:

[source,bash]
----
python3 tools/quality_gate.py lint
python3 tools/quality_gate.py arch-lint
python3 tools/quality_gate.py arch-tests
python3 tools/quality_gate.py typecheck
python3 tools/quality_gate.py test
----

The gate runs Ruff, import-linter, architecture tests, mypy, and unittest
discovery against the `src/tiny_swarm_world` source tree and root `tests/`
directory.

== Main Python Entrypoint

The main workflow entry point is:

[source,bash]
----
PYTHONPATH=src python3 -m tiny_swarm_world
----

Running without arguments does not execute infrastructure commands. List the
workflow-level commands first:

[source,bash]
----
PYTHONPATH=src python3 -m tiny_swarm_world --list-workflows
----

Static preflight remains available without live-infrastructure consent:

[source,bash]
----
PYTHONPATH=src python3 -m tiny_swarm_world --preflight
----

Non-mutating workflow checks can be selected explicitly:

[source,bash]
----
PYTHONPATH=src python3 -m tiny_swarm_world platform verify
----

The safe setup probe is `setup run` without `--live`. It refuses before setup
services are constructed:

[source,bash]
----
PYTHONPATH=src python3 -m tiny_swarm_world setup run
----

The canonical live operator command requires live-infrastructure consent before
services are constructed and uses `lxc_native` by default:

[source,bash]
----
PYTHONPATH=src python3 -m tiny_swarm_world setup run --live
----

The setup sequence includes `platform expose`, which configures the
LXC-native manager gateway for published Swarm service ports after platform
init/reconcile and before deployment bootstrap.

Select a managed LXC backend explicitly when both LXD and Incus are available:

[source,bash]
----
PYTHONPATH=src python3 -m tiny_swarm_world --lxc-backend lxd setup run --live
PYTHONPATH=src python3 -m tiny_swarm_world --lxc-backend incus setup run --live
----

The command prompts for a short confirmation:

[source,text]
----
Live setup can change local infrastructure. Continue? [y/N]:
----

Answer `y` only when changing local infrastructure is intentional. Without
`--live` and the short confirmation, the entry point reports
`REFUSED_LIVE_CONSENT_MISSING` before application services are constructed.

Destructive platform workflows also require the exact confirmation phrase:

[source,bash]
----
PYTHONPATH=src python3 -m tiny_swarm_world platform reset --confirm RESET_TINY_SWARM_PLATFORM --live
PYTHONPATH=src python3 -m tiny_swarm_world platform destroy --confirm DESTROY_TINY_SWARM_PLATFORM --live
----

Reset and destroy are explicit confirmation contracts. Their destructive
execution is limited to configured managed LXC-native nodes whose Tiny Swarm
World ownership evidence is verified before mutation. Already absent managed
nodes count as successful evidence. Missing, mismatched, or unmanaged
ownership evidence blocks before deletion.

Artifact and deployment workflow command names are reserved by the CLI contract
and dispatch to guarded workflow contracts on the default `lxc_native` path.
They require live consent for mutating phases and report blocked or failed
phase evidence when live Nexus, registry, Portainer, Swarm, external-input, or
observed-state prerequisites are unavailable.

Optional live smoke validation is a separate operator action, not part of
`python3 tools/quality_gate.py quality`. Run `setup run --live` only on a
disposable or recoverable local environment and only when LXD/Incus/LXC or
provider state, Docker Swarm, networking, Portainer,
Nexus, Jenkins, RabbitMQ, SonarQube, Swagger/NGINX, image, and stack changes
are intentional.

== Verification Scope

The workflow safety behavior is verified through unit tests, architecture
checks, import-linter contracts, type checks, and static command catalog
validation. The default quality gate does not run live LXD, Incus, LXC
container lifecycle, Docker Swarm, compose, netplan, socat,
Portainer, Nexus, Jenkins, RabbitMQ, SonarQube, or Swagger/NGINX commands.

== Service Preparation

The workflow-level CLI is the supported setup surface. Portainer, Nexus, image
publication, stack deployment, and final verification belong to setup
boundaries. On the default `lxc_native` path they run only through guarded
artifact, deployment, external-input, and observed-state contracts and report
blocked or failed phase evidence when live prerequisites are unavailable:

[source,bash]
----
PYTHONPATH=src python3 -m tiny_swarm_world setup run --live
----

Former direct service preparation helpers under `infra/prepare` have been
retired and are not development quality-gate checks.

== Legacy And Direct Scripts

The workflow-level CLI is the supported operator entry point. Former direct
scripts under `infra/prepare` and host-side orchestration scripts under
`infra/compose` have been retired. Direct scripts under `infra/swarm` are
legacy live operations and are not protected by the CLI consent gate.
The canonical classification is maintained in
`documentation/system/live-operation-surfaces.adoc`.

[options="header"]
|===
| Path | Status
| `src/tiny_swarm_world/__main__.py` | Supported workflow-level entry point with live-consent and confirmation contracts.
| `infra/prepare/**` | Retired former direct service preparation surface; no executable setup helpers are kept there.
| `infra/config/compose/**/docker-compose.yml` | Supported stack assets, not standalone quality-gate commands.
| `infra/compose/**/Dockerfile` | Supported image source assets used by the Python setup workflow.
| `infra/swarm/**` | Legacy live-infrastructure surface; not a supported workflow entry point.
|===

Do not re-enable legacy or retired scripts without a dedicated cleanup slice,
tests, and rollback notes.

== Nexus Bootstrap

Nexus bootstrap is owned by the workflow-level setup command. Configuration can
be supplied with environment variables where supported by the Python adapters:

Important variables:

* `TSW_PORTAINER_URL`
* `TSW_PORTAINER_USERNAME`
* `TSW_PORTAINER_PASSWORD`
* `TSW_PORTAINER_ENDPOINT`
* `TSW_NEXUS_STACK_NAME`
* `TSW_NEXUS_URL`
* `TSW_NEXUS_ADMIN_USERNAME`
* `TSW_NEXUS_ADMIN_PASSWORD`
* `TSW_NEXUS_INITIAL_PASSWORD_PATH`
* `TSW_NEXUS_MAX_ATTEMPTS`
* `TSW_NEXUS_WAIT_SECONDS`

The repository still contains local static defaults for this non-production
system. For real local work, provide explicit values through environment
variables where supported.
Do not copy local compatibility defaults into new configuration or logs.

== Stack Deployment

Stack deployment is owned by the workflow-level setup boundary. On the default
`lxc_native` path it runs only through guarded stack, external-input, and
observed-state contracts, and live success still requires explicit deployment
and readiness evidence. Direct Portainer upload scripts have been retired so
compose assets cannot bypass the CLI consent gate.

== Service Access Dashboard And Vaultwarden

The full guided setup selects the service-access stack profile by default. It
provides the intended management landing page contract for service links and
Vaultwarden credential references after a verified provider-specific live
deployment. Operators who intentionally want only the older base service set
can pass `--service-profile default` to setup, deployment, or preflight
commands.

Implemented repository assets:

* `infra/config/compose/service-access/docker-compose.yml` defines the
  service-access dashboard, Vaultwarden and service-access NGINX services.
* `infra/compose/service-access/dashboard/**` contains a static dashboard
  image asset.
* `infra/compose/service-access/nginx/**` contains the image-packaged NGINX
  route configuration.

When the default full profile is deployed by an operator workflow, the
dashboard route is `http://localhost` and the Vaultwarden route is `8086`.
The dashboard is a static management table with `Server`, `URL`, `user` and
`pwd` columns. It provides central paths such as `/jenkins`, `/nexus`,
`/portainer`, `/rabbitmq`, `/sonarqube`, `/swagger` and `/vaultwarden`.
It does not show password values.

Authorized operators reveal or copy password values only through
Vaultwarden's authenticated UI.

The current central `http://localhost` route requires provider-specific
redeployment before browser verification.

== Commit And Push Shortcut

For Codex-assisted repository maintenance, the user command `cc` executes the
workflow documented in `commit_push.md`: repository inspection, quality gate,
