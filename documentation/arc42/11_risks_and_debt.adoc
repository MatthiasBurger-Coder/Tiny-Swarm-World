==  Risks & Technical Debt

Potential challenges and risks.

=== Remaining Command Execution Risk

The command catalog classifies command metadata and enforces workflow
allow-lists before command building when product command YAML exists. The
current Multipass-removal baseline intentionally has no product
`command_*.yaml` files. Retained VM-named command-builder helpers are legacy
command-template infrastructure and must not be treated as supported Multipass
provider behavior. Shell command replacement with argv-style execution remains
future hardening work if a command catalog is reintroduced.

Command parameters are constrained by type-specific patterns before
substitution. Runner logs no longer include the full raw command string,
stdout, or stderr payloads. Verification evidence rejects raw or sensitive keys
and values before persistence.

Remaining debt:

* the runner still uses shell execution for command-template infrastructure
  when a catalog is supplied;
* no current product command catalog exists, so any reintroduced catalog must
  restore non-vacuous command-backed verification coverage;
* composed mutating platform workflows now block with explicit target IDs and
  operator-facing reasons before apply, but still need concrete command-backed
  verification contracts before they can safely execute live steps;
* the host-neutral desired inventory baseline is committed and aligned with
  the default service-access setup profile, but observed inventory collection
  and reconciliation are not yet wired into platform verification workflows;
* removed legacy scripts remain historical live-operation surfaces outside the
  workflow-level CLI consent guard;
* reset/destroy teardown is wired for configured managed LXC-native nodes, but
  live operator validation, retention hardening, and unmanaged-resource
  recovery guidance remain open;
* artifact and deployment workflows are now explicit composition contracts;
  artifact workflows have tested Nexus repository and setup image-publication
  contracts, but default CLI composition still needs live registry checks and
  observed-state verification hardening;
* `artifacts prepare` and `deployment apply` are live-consent gated before
  service construction, but the approved live behavior behind those gates is
  still technical debt.

=== Autonomous Setup Safety Debt

The autonomous setup safety contract is accepted. The canonical `setup run`
workflow-level orchestrator now exists and fails closed behind live consent,
but end-to-end full runnable setup remains incomplete.

Remaining debt:

* `setup run` now includes the service-access management stack in the full
  profile, but still needs complete end-to-end verification hardening;
* setup preflight and setup manifest contracts exist, but still need live
  profile hardening for resource, credential, and readiness handoff detail;
* command-backed verification remains incomplete for live Platform phases;
* Artifacts readiness still needs image publication and registry observed-state
  contracts; Deployment readiness has default stack-presence contracts but
  still needs container and service health observed-state contracts;
* Portainer, Nexus, Jenkins, RabbitMQ, SonarQube, and Swagger/NGINX service
  readiness must not be claimed until verified through tests or explicit live
  smoke evidence;
* direct preparation scripts and host-side orchestration scripts have been
  removed, and legacy helper reintroduction still needs
  migration or clear legacy retirement;
* automatic host package installation, non-interactive live consent, and direct
  script promotion need future ADRs before they can change the accepted safety
  model.

=== LXC Native Provider Migration Risks

The LXC-native provider direction is accepted and partially implemented.
Default provider selection, LXD/Incus readiness, node lifecycle adapters,
Docker Engine setup inside managed LXC nodes, Swarm bootstrap, setup/platform
init integration, LXC-native reconcile no-op completion, and unsupported
provider rejection are implemented and test-backed.

Remaining risks and debt:

* default artifact and deployment workflows have provider-native publication,
  stack, and observed-state contracts, but live registry, Portainer, Swarm,
  external-input, and readiness evidence still needs explicit operator
  validation;
* stale direct instance-level `tsw-proxy-*` devices from older runs can still
  block reset or setup until the operator runs the explicit
  `platform repair-lxc-proxy-drift` workflow or manually resolves the named
  provider state;
* Docker Swarm inside LXD/Incus containers has not been live-validated by this
  repository workflow;
* WSL2 success could be overclaimed if documentation, static tests, or native
  Linux evidence are treated as WSL2 live evidence;
* Docker Swarm inside LXD/Incus containers may require nesting, cgroup,
  AppArmor, seccomp, capability, privileged-profile, or host-mount decisions
  that need security and DevOps review;
* automatic provider installation, daemon repair, group membership changes,
  bridge/firewall mutation, and non-interactive live consent remain outside
  the accepted default behavior until later ADRs or explicit operator approval
  cover them.

=== Service Access Dashboard And Vaultwarden Baseline Risks

The service-access dashboard and Vaultwarden baseline is implemented as
repository compose assets, image-packaged dashboard/NGINX assets, setup
profile contracts, setup manifest requirements and static tests.
The current central `http://localhost` route requires a fresh
provider-specific redeployment before it is claimed as verified.

Remaining risks and debt:

* credential leakage could occur through dashboard content, compose defaults,
  logs, evidence, screenshots or exported artifacts if password values are
  duplicated outside Vaultwarden;
* required setup password values must remain operator-supplied; committed
  static defaults are not accepted as preflight secret-value sources;
* Vaultwarden admin-token ownership, rotation, persistence, backup, restore
  and rollback are not implemented and remain blockers before production-like
  claims; the committed default is only the external Swarm secret name, never
  the token value;
* missing Vaultwarden external Swarm input now blocks before service-access
  stack deployment, but creating or rotating that input remains an operator
  responsibility outside the default quality gate;
* a bootstrap cycle is possible if required setup credentials are moved into
  Vaultwarden before Vaultwarden itself is reachable;
* service reachability could be falsely presented as healthy without fresh
  observed-state or HTTP evidence; current dashboard assets render catalog
  reachability as unknown unless evidence is collected separately;
* the central `http://localhost` service-access route and local `https://localhost` Vaultwarden route are implemented in
  repository assets but still needs a fresh live redeployment and browser smoke
  test before it can be claimed as verified;
* local registry availability and image publication still need live
  observed-state hardening beyond the static artifact contracts;
* plain HTTP access is acceptable only as a local development baseline until a
  later ADR accepts public certificate automation or wider exposure controls;
* scope could drift into a React/browser frontend project or Traefik routing
  without the required ADR and quality-gate coverage.

Resolved or narrowed risks:

* the published port collision with Swagger/NGINX port `80` is resolved by
  making service-access the owner of `80` and moving Swagger/NGINX to `8084`;
* normal proxy exposure no longer creates direct instance-level `tsw-proxy-*`
  devices; expected proxy state belongs to the manager-only LXC profile;
* direct instance-level project proxy devices remain drift and are repairable
  only through the explicit profile-equivalence-gated repair workflow;
* dashboard and NGINX repository assets are image-packaged for the
  Portainer-managed stack path instead of bind-mounted from the repository;
* static tests reject React/Vite project files, live orchestration helpers,
  credential-bearing dashboard links, and raw password rendering.
* service-access live evidence must be collected separately for the default
  `lxc_native` provider baseline.
