==  Risks & Technical Debt

Potential challenges and risks.

=== Remaining Command Execution Risk

The command catalog now classifies command metadata and enforces workflow
allow-lists before command building. The underlying runner still uses shell
execution for the existing command templates because the current operational
catalog is shell-oriented. Shell command replacement with argv-style execution
remains future hardening work and must preserve current Linux/WSL behavior.

Command parameters are constrained by type-specific patterns before
substitution. Runner logs no longer include the full raw command string,
stdout, or stderr payloads. Verification evidence rejects raw or sensitive keys
and values before persistence.

Remaining debt:

* the runner still uses shell execution for the existing shell-oriented command
  catalog;
* command catalog entries mostly declare manual verification, so command-backed
  verification remains planned work;
* composed mutating platform workflows now block with explicit target IDs and
  operator-facing reasons before apply, but still need concrete command-backed
  verification contracts before they can safely execute live steps;
* the host-neutral desired inventory baseline is committed, but observed
  inventory collection and reconciliation are not yet wired into platform
  verification workflows;
* legacy scripts under `infra/swarm` remain live-operation surfaces outside the
  workflow-level CLI consent guard;
* reset/destroy retention and teardown semantics are not wired yet;
* artifact and deployment workflows are now explicit composition contracts;
  artifact workflows have tested Nexus repository contracts, but default CLI
  composition remains blocked until live artifact steps, image publication,
  registry checks, and observed-state verification are wired deliberately;
* `artifacts prepare` and `deployment apply` are live-consent gated before
  service construction, but the approved live behavior behind those gates is
  still technical debt.

=== Autonomous Setup Safety Debt

The autonomous setup safety contract is accepted. The canonical `setup run`
workflow-level orchestrator now exists and fails closed behind live consent,
but end-to-end full runnable setup remains incomplete.

Remaining debt:

* `setup run` does not yet install and verify the full runnable profile;
* setup preflight and setup manifest contracts exist, but still need live
  profile hardening for resource, credential, and readiness handoff detail;
* command-backed verification remains incomplete for live Platform phases;
* Artifacts readiness still needs image publication and registry observed-state
  contracts; Deployment readiness has default stack-presence contracts but
  still needs container and service health observed-state contracts;
* Portainer, Nexus, Jenkins, RabbitMQ, SonarQube, and Swagger/NGINX service
  readiness must not be claimed until verified through tests or explicit live
  smoke evidence;
* direct scripts under `infra/prepare` and host-side orchestration scripts
  under `infra/compose` have been retired, while `infra/swarm` still needs
  migration or clear legacy retirement;
* automatic host package installation, non-interactive live consent, and direct
  script promotion need future ADRs before they can change the accepted safety
  model.

=== Service Access Dashboard And Vaultwarden Baseline Risks

The service-access dashboard and Vaultwarden baseline is accepted as a
requirement and security direction, but implementation has not started.

Remaining risks and debt:

* credential leakage could occur through dashboard content, compose defaults,
  logs, evidence, screenshots or exported artifacts if password values are
  duplicated outside Vaultwarden;
* Vaultwarden admin-token ownership, rotation, persistence, backup, restore
  and rollback are not implemented and remain blockers before production-like
  claims;
* a bootstrap cycle is possible if required setup credentials are moved into
  Vaultwarden before Vaultwarden itself is reachable;
* service reachability could be falsely presented as healthy without
  observed-state or HTTP evidence;
* service-access routing can collide with the existing Swagger/NGINX port `80`
  binding unless a later slice defines shared ingress or a non-conflicting
  published port;
* Portainer stack upload currently sends compose content, so mounted
  dashboard or NGINX assets need image packaging or explicit asset
  preparation before deployment;
* plain HTTP access is acceptable only as a local development baseline until a
  later ADR accepts TLS, certificate automation or wider exposure controls;
* scope could drift into a React/browser frontend project or Traefik routing
  without the required ADR and quality-gate coverage.
