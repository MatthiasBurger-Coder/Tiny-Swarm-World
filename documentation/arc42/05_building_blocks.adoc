== Building Blocks

Tiny Swarm World now separates three repository roles explicitly.

=== Python Automation

`src/tiny_swarm_world` contains the Python orchestration code.

* `domain` contains core command, VM, network, deployment, and Nexus concepts.
* `application` contains use cases, command orchestration services, and ports.
* `infrastructure` contains adapters for command execution, HTTP clients,
  repositories, YAML handling, dependency wiring, logging, and local file
  access.

The canonical development import root is `src`, and commands that execute the
Python package use `PYTHONPATH=src`.

`src/tiny_swarm_world/infrastructure/composition.py` is the local composition
root for the Python automation. It wires concrete infrastructure adapters into
application services and keeps `src/tiny_swarm_world/__main__.py` thin. Services
such as `VmIpList`, `MultipassInitVms`, and `NetworkPrepareNetplan` are
constructed with explicit ports and repositories instead of creating
infrastructure adapters inside the application layer.

The composition root now exposes separate in-process application service
bundles. These bundles are responsibility boundaries inside the Python
automation process; they are not independently deployable services or
microservices:

* `PlatformServices` contains the current platform automation for Multipass,
  network setup, Docker installation, Swarm initialization, preflight checks,
  VM IP discovery, socat management, and workflow objects for platform
  `init`, `reconcile`, `verify`, `reset`, and `destroy` commands.
* `ArtifactServices` is the explicit artifacts boundary. It owns
  `artifacts prepare` and `artifacts verify` workflow contracts. The default
  composition remains fail-closed, while the workflow classes can now execute
  configured Nexus repository and registry-readiness checks only after each
  step exposes typed verification evidence.
* `DeploymentServices` is the explicit deployment boundary. The deployment
  service namespace now owns `EnsurePortainerStack`, `EnsureNexusStack`, and
  the generic `EnsureServiceStack` contract for the base runnable-service
  stack set plus the service-access setup profile.
  `EnsurePortainerStack` is a post-bootstrap Portainer API contract for
  compose lookup plus stack create/update through `PortComposeFileRepository`
  and `PortPortainerClient`. The composition bundle does not construct a
  Portainer client or run a stack mutation during CLI dispatch in the current
  slice. `deployment apply` and `deployment verify` can evaluate configured
  stack contracts, but live success still depends on live consent, stack
  bootstrap and observed service readiness.
* `ApplicationServices` aggregates `platform`, `artifacts`, and `deployment`
  instead of aliasing `PlatformServices`.

The service-access profile is implemented as a Deployment stack contract, not
as a new microservice and not as a React frontend project. The full guided
setup selects `ServiceStackProfile.SERVICE_ACCESS` by default and adds
`SERVICE_ACCESS_STACK_CONTRACT` with the services
`service-access-dashboard`, `vaultwarden`, and `service-access-nginx`. The
base stack profile remains available for explicit `--service-profile default`
runs. The service-access setup manifest adds ports `8085` and `8086` plus the
`TSW_VAULTWARDEN_ADMIN_TOKEN_SECRET` credential-source name only.

The service-access dashboard and NGINX assets are image-packaged under
`infra/compose/service-access/**`, while the stack definition lives under
`infra/config/compose/service-access/docker-compose.yml`. Static tests verify
that the dashboard links only to Vaultwarden, does not render password values,
does not introduce React/Vite files, and keeps reachability unknown until
observed evidence is available.

The default Artifacts and Deployment workflow bundles are intentional
fail-closed building blocks. They create stable composition boundaries while
service ownership moves in dedicated workflow slices, and they keep later live
Nexus, registry, image publication, Portainer and service-access work behind
explicit contracts.
Compatibility properties on `ApplicationServices` keep the existing
service-level accessors available for migration, but the public Python
entrypoint now routes through workflow-level commands.

Console/status UI is also an in-process responsibility surface. It owns
terminal status rendering, progress feedback and command-runner status
presentation. It is not a browser React frontend.

=== Autonomous Setup Boundary

The autonomous setup workflow is an implemented fail-closed orchestration over
the existing in-process boundaries. It is not a new deployable service,
microservice, Spring Boot application, Java responsibility, or browser
frontend.

The setup boundary coordinates Platform, Artifacts, Deployment, Shared, and
Console/status UI contracts:

* Platform owns host preflight, Multipass, Docker installation, Docker Swarm,
  network, VM, and resource readiness behavior.
* Artifacts owns Nexus, registry, image publication, and repository readiness
  behavior.
* Deployment owns Portainer, compose assets, service stack apply behavior, and
  service readiness verification.
* Shared owns command abstractions, local evidence storage, redaction,
  configuration loading, and composition wiring.
* Console/status UI owns terminal-only setup progress and recovery output.

The current repository state includes the canonical `setup run` command,
live-consent refusal before setup service construction, phase sequencing, and
terminal recovery states. Lower Platform, Artifacts, and Deployment live
contracts still need complete observed-state evidence before documentation can
claim that the setup path installs a full runnable system.

=== Typed Command Catalog

Command YAML files under `infra/config/**/command_*.yaml` are product
configuration and are loaded as typed command catalog entries.

The domain command model owns parser-independent metadata and safety rules:

* stable command `id`
* command `intent`
* `execution_mode`
* `safety_class`
* owned `scope`
* `allowed_workflows`
* template `parameters`
* declared `effects`
* `verify` specification

The infrastructure YAML repository parses files and maps them to domain
command entities. Application command building receives an explicit workflow
identifier and rejects commands whose `allowed_workflows` do not include the
active workflow before executable command strings are built.

The supported safety classes are `safe_read`, `safe_mutation`, `destructive`,
`credential_mutation`, and `network_mutation`. Destructive command patterns
such as Multipass delete or purge are accepted only when classified as
`destructive` and limited to reset or destroy workflows. Mutating commands
must declare a verification specification. Commands that declare
`runtime_change` cannot be classified as `safe_read`. Commands that emit
credential material must be classified as `credential_mutation` and must
declare an evidence policy that redacts output and forbids raw output storage.

=== Inventory And Verification Evidence

The domain now contains parser-independent inventory and verification concepts
under `tiny_swarm_world.domain.inventory`.

* `DesiredInventory` and `VmDesiredState` describe intended product state.
* `ObservedInventory`, VM, network, Docker, Swarm, stack, and artifact registry
  observed-state types describe runtime facts collected by adapters.
* `VerificationResult` and `VerificationStatus` model typed verification
  evidence separately from raw command output.

Application repository ports define the desired inventory, observed inventory,
and verification evidence contracts. Infrastructure adapters own YAML parsing,
JSON persistence, and path validation. Observed inventory and verification
evidence use dedicated local-state adapters instead of the generic file
manager so raw evidence is not logged and file lookup cannot drift into
`infra/config`.

The default desired inventory baseline is committed at
`infra/config/inventory/desired_inventory.yaml`. It records host-neutral VM
names, roles, resource sizes, expected stacks, and artifact registries. It must
not contain host-specific IP addresses, gateway values, user names, tokens,
passwords, or secrets.

=== Infrastructure Assets

`infra` contains infrastructure and deployment assets that are not Python
application code.

* `infra/config` contains command, VM, network, and compose configuration.
* `infra/compose` contains image build-context assets.
* `infra/prepare` contains retired notes for former direct service preparation
  helpers and must not contain executable setup entry points.
* `infra/swarm` contains legacy Swarm and Multipass helper scripts. This area
  is quarantined as legacy live-infrastructure surface, not a supported
  workflow entry point. It must not run in default quality gates, and files
  should be removed or migrated only in an explicit cleanup slice with tests
  and rollback notes.

Live infrastructure scripts can modify VMs, networking, Docker Swarm state,
service stacks, and credentials. They are not part of the normal development
quality gate.

Direct live-operation scripts must not become the canonical setup path by
documentation alone. Former direct service preparation helpers and host-side
compose orchestration scripts are retired; new setup behavior must go through
Python ports, adapters, tests, verification, and live-consent controls.
