== Runtime View

Processes and workflows during execution.

=== Application Composition

`src/tiny_swarm_world/__main__.py` remains a thin entry point. It parses
operator arguments, lists workflow-level commands without constructing
services, checks live-infrastructure consent for mutating workflows, verifies
destructive workflow confirmation before service construction, builds
the required service bundle through
`src/tiny_swarm_world/infrastructure/composition.py`, and delegates selected
commands to composed workflow objects. Platform commands build the full
`ApplicationServices` aggregate. Artifact and Deployment commands build only
their own boundary service bundle so they cannot fail on unrelated platform
configuration before returning their explicit contract result.

The composition root builds `ApplicationServices` as an aggregate of three
service bundles:

* `platform` for current VM, network, Docker, Swarm, preflight, VM IP, and
  socat automation.
* `artifacts` for artifact registry and publishing workflow contracts.
* `deployment` for stack lifecycle behavior, starting with Nexus stack
  create/update ownership in the application service namespace.

At this stage, Artifacts and Deployment are composition boundaries inside the
same Python automation process, not independently deployable microservices.
Their bundles are side-effect free when constructed. The public CLI no longer
dispatches low-level service names through `--run`; transitional
`ApplicationServices` properties remain only for migration and internal
compatibility. Artifact and deployment workflow command names are declared by
the CLI contract and dispatch to blocked workflow objects with stable reasons
until later slices wire their runtime behavior. Mutating boundary workflows
such as `artifacts prepare` and `deployment apply` still require live consent
before their boundary service bundle is constructed, then return `blocked`
because Nexus, registry, and Portainer contracts are not wired yet.

=== Artifact And Deployment Workflow Contract Flow

The current artifact and deployment runtime flow is deliberately fail-closed:

* `artifacts prepare` requires live consent, dispatches to
  `ArtifactPrepareWorkflow`, and returns `blocked` until image build, image
  push, and Nexus repository setup contracts exist.
* `artifacts verify` dispatches without live consent and returns `blocked`
  until artifact registry observed-state verification is implemented through
  an application port.
* `deployment apply` requires live consent, dispatches to
  `DeploymentApplyWorkflow`, and returns `blocked` until Portainer stack apply
  and command-backed verification contracts exist.
* `deployment verify` dispatches without live consent and returns `blocked`
  until stack and service observed-state verification is implemented through
  deployment ports.

No Portainer, Nexus, Docker registry, image build, image push, or stack
deployment side effect is run by these workflow contracts.

=== Inventory And Evidence Runtime Flow

Desired inventory is a committed product configuration concern and can be read
from `infra/config` through an application repository port. Observed inventory
and verification evidence are local runtime facts. They are persisted by
infrastructure adapters under `.tiny-swarm-world/` and are not committed.

The local observed-state adapters resolve every target path under the repository
local-state root and reject absolute paths, traversal outside that root, and
targets under `infra/config`. Verification evidence stores typed summaries
through `VerificationResult`; raw command strings, stdout, stderr, environment
payloads, and keys that imply secrets are rejected before persistence.

No live Multipass, Docker Swarm, netplan, socat, Portainer, Nexus, Jenkins,
RabbitMQ, SonarQube, compose, or stack deployment command is required to create
or round-trip these inventory artifacts in tests.

=== Verify After Apply Runtime Gate

Mutating platform workflows now execute application-level steps through an
apply-then-verify gate. A mutating step must expose a verification path before
the workflow runs it. If that path is missing, the workflow returns `blocked`
and does not apply the step.

The gate itself is implemented as fail-closed workflow behavior. At the current
system-unification baseline, the composed `platform init` and
`platform reconcile` steps expose explicit verification target IDs and
operator-safe block reasons, but they still do not expose concrete
command-backed verification contracts. These workflows therefore return
`blocked` before running live steps. This keeps Multipass, Docker Swarm,
netplan, and VM reconciliation commands from running until later slices add
tested verification wrappers or equivalent verifiable step contracts.

Continuation requires both parts:

* apply succeeds;
* verify returns `VerificationStatus.verified` evidence.

Apply exceptions and explicit failed apply results return
`failed_to_apply`. Verification exceptions, failed verification results, or
non-verified evidence return `failed_to_verify`, while missing evidence returns
`blocked`. Later mutating steps are not run after any of these outcomes.

When a verification evidence repository is configured, the workflow appends
the typed verification result before making the continuation decision.

The non-mutating `platform verify` workflow also interprets preflight results.
Failed preflight checks now return `failed_to_verify` with summary-only
verification evidence instead of reporting a completed verify workflow.
