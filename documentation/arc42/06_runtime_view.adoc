== Runtime View

Processes and workflows during execution.

=== Application Composition

`src/tiny_swarm_world/__main__.py` remains a thin entry point. It parses
operator arguments, lists workflow-level commands without constructing
services, checks live-infrastructure consent for mutating workflows, verifies
destructive workflow confirmation before service construction, builds
the required service bundle through
`src/tiny_swarm_world/infrastructure/composition.py`, and delegates selected
commands to composed workflow objects. Platform commands build the full
`ApplicationServices` aggregate. Artifact and Deployment commands build only
their own boundary service bundle so they cannot fail on unrelated platform
configuration before returning their explicit contract result. `setup run`
builds a dedicated `SetupServices` bundle only after accepted live consent and
delegates all sequencing to the setup application workflow.

The composition root builds `ApplicationServices` as an aggregate of three
service bundles:

* `platform` for LXC-native provider selection, LXD/Incus readiness, managed
  node lifecycle, network, Docker, Swarm, preflight, node address, and socat
  automation.
* `artifacts` for artifact registry and publishing workflow contracts.
* `deployment` for stack lifecycle behavior, starting with Nexus stack
  create/update ownership in the application service namespace.

Provider migration status: LXC-native through LXD or Incus is the implemented
default selection and setup/platform init direction. The runtime now has
provider-neutral contracts, LXD/Incus readiness checks, node lifecycle
adapters, Docker Engine setup and Swarm bootstrap adapters, setup integration,
LXC proxy exposure, explicit stale proxy drift repair, and fail-closed
unsupported-provider behavior. Default platform reconcile is currently a
verified no-op boundary for `lxc_native` after Platform init. Default platform
expose uses `swarm-manager` as the host-reachable LXC gateway for published
service ports by ensuring profile-level proxy devices in the manager-specific
`docker-swarm-manager` profile. The explicit
`platform repair-lxc-proxy-drift` workflow is separate from normal setup and
removes stale direct manager instance proxy devices only after the equivalent
manager-profile device is verified. Artifact and deployment workflows use
configured LXC-native contracts for Nexus, registry, image publication, stack
deployment, external inputs, and service readiness. They return blocked or
failed evidence when a live prerequisite is not observable. No documentation
may claim Docker Swarm-in-container or WSL2 live success without
target-specific evidence.

At this stage, Artifacts and Deployment are composition boundaries inside the
same Python automation process, not independently deployable microservices.
Their bundles are side-effect free when constructed. The public CLI no longer
dispatches low-level service names through `--run`; transitional
`ApplicationServices` properties remain only for migration and internal
compatibility. Artifact and deployment workflow command names are declared by
the CLI contract. Mutating boundary workflows such as `artifacts prepare` and
`deployment apply` require live consent before their boundary service bundle is
constructed, then execute only configured contracts with verify-after-prepare
or verify-after-apply evidence. Missing registry, Portainer, Swarm,
external-input, or readiness evidence returns explicit blocked or failed phase
states.

Setup orchestration is also an in-process composition concern. It is not a new
microservice boundary and it does not construct low-level adapters in the entry
point. The setup bundle wires phase calls for preflight, platform init,
platform reconcile, platform expose, deployment bootstrap, artifacts prepare,
artifacts verify, deployment apply, deployment verify, and final platform
verify. `platform expose` configures the LXC-native manager gateway only; it
does not proxy individual service containers or worker nodes. It does not wire
platform reset or destroy behavior; the `install.sh` fresh-install wrapper owns
the reset prelude before it invokes setup.

=== Artifact And Deployment Workflow Contract Flow

The current artifact and deployment runtime flow is deliberately fail-closed:

* `artifacts prepare` requires live consent and dispatches to
  `ArtifactPrepareWorkflow`. The workflow can run configured artifact steps
  only when each step provides verify-after-prepare evidence. Missing live
  Nexus, registry, image build, or image push evidence returns blocked or
  failed artifact evidence.
* `artifacts verify` dispatches without live consent and can evaluate
  configured artifact verification checks. Missing registry observed-state
  evidence returns blocked verification evidence.
* `deployment bootstrap` starts the Portainer stack, initializes admin access,
  registers the local Docker endpoint in Portainer, and then prepares the
  Nexus stack. Endpoint registration is a bootstrap contract because later
  application stacks use Portainer as the deployment gateway.
* `deployment apply` requires live consent, dispatches to
  `DeploymentApplyWorkflow`, and can run configured stack apply contracts only
  when they expose verify-after-apply evidence. The tested
  `EnsureServiceStack` contract can create or update Portainer-managed stacks
  and then verify stack registration through `PortPortainerClient`. The
  default `lxc_native` path wires Portainer-admin bootstrap, stack contracts,
  service-access external-input checks, and service-readiness verification
  through deployment workflow ports.
* `deployment verify` dispatches without live consent and returns `blocked`
  or `failed_to_verify` when observed stack or service-readiness evidence is
  missing. It does not claim container or service health from endpoint
  configuration alone.

Nexus repository mutation, Docker registry mutation, image build, and image
push side effects run only through live-consent-gated workflow execution, not
through service construction or static verification. Nexus repository and
Portainer API behavior is covered by isolated adapter and application-service
tests with fakes; live service readiness remains separate observed-state
evidence and must not be inferred from endpoint configuration alone.

Portainer admin initialization has a fail-fast rejection gate. Deployment
automation may retry transient initialization errors while the service is
becoming ready, but once a Portainer admin endpoint rejects initialization and
the requested credentials still cannot authenticate, infrastructure adapters
translate that response into the typed application-port rejection
`PortainerAdminInitializationRejected`. If the requested credentials become
active after the rejection, the application deployment service treats the
target state as achieved; otherwise it re-raises the typed rejection instead
of exhausting readiness retries. This keeps operator-visible failures prompt
while preserving retries for genuine transient readiness errors and avoiding a
false success for wrong credentials.

=== Console Status Runtime Flow

Console/status UI remains terminal-only. The application UI port now owns the
shared status vocabulary used by command-runner UI adapters and terminal
renderers. Command runners report per-instance terminal results with the same
`Success`, `Error`, and `Failed` semantics that the Linux console adapter uses
to decide when all visible instances have reached a terminal state.

Aggregate updates use the reserved `all` instance and are stored separately
from per-VM state. The terminal renderer derives completion summaries from
both aggregate and per-instance status, so failed command runs display
`All instances completed with errors` instead of the ambiguous success-only
completion message.

Setup-oriented terminal status extends the same vocabulary without adding a
browser UI. `Refused`, `Blocked`, `Resource gated`, `Failed to prepare`,
`Failed to apply`, `Failed to verify`, `Failed`, and `Not run` are terminal
non-success states. The UI normalizes spacing, underscores, and hyphens for
these states, keeps later `Success` updates from overwriting an existing
non-success setup state, and renders a textual `Action:` recovery action when a
known recovery path exists. Successful setup-style terminal values such as
`completed`, `passed`, and `verified` remain terminal success states and do
not render recovery guidance.

When an aggregate setup state is terminal while individual rows are still
pending, the Linux terminal renderer shows the aggregate status and recovery
guidance explicitly. This avoids relying on color and prevents refused,
blocked, resource-gated, or failed states from being presented as success.

No browser frontend, React component, JavaScript build tool, TSX/JSX file, or
browser state is part of this console/status runtime flow.

Installation observability is implemented for the current declared setup,
platform, and command-executor runtime path. Command-runner UI adapters, the
shared terminal status vocabulary, structured workflow progress, and
cross-cutting method trace events are wired through ports and infrastructure
adapters.

ADR `documentation/architecture/adr-cross-cutting-method-trace-logging.adoc`
records that logging is a cross-cutting Shared observability module. Covered
installation runtime methods emit method entry, normal exit, and exception
exit trace events through the application trace port. Composition wires the
same safe event stream to centralized logging and terminal `PortUI`
projections while keeping domain and application code free from concrete UI
and logger configuration dependencies.

`setup run` starts terminal status rendering only after accepted live consent.
The entry point delegates this lifecycle to infrastructure composition; it does
not construct concrete UI adapters, start terminal threads, or update aggregate
terminal status directly.

=== Inventory And Evidence Runtime Flow

Desired inventory is a committed product configuration concern and can be read
from `infra/config` through an application repository port. Observed inventory
and verification evidence are local runtime facts. They are persisted by
infrastructure adapters under `.tiny-swarm-world/` and are not committed.

The local observed-state adapters resolve every target path under the repository
local-state root and reject absolute paths, traversal outside that root, and
targets under `infra/config`. Verification evidence stores typed summaries
through `VerificationResult`; raw command strings, stdout, stderr, environment
payloads, and keys that imply secrets are rejected before persistence.

No live LXD, Incus, LXC container lifecycle, Docker Swarm, netplan,
socat, Portainer, Nexus, Jenkins, RabbitMQ, SonarQube, compose, or stack
deployment command is required to create or round-trip these inventory
artifacts in tests.

=== Verify After Apply Runtime Gate

Mutating platform workflows now execute application-level steps through an
apply-then-verify gate. A mutating step must expose a verification path before
the workflow runs it. If that path is missing, the workflow returns `blocked`
and does not apply the step.

The gate itself is implemented as fail-closed workflow behavior. Default
`platform init` now uses LXC-native node-provider steps with verification
target IDs and LXD/Incus provider readiness guards, then continues to
container Docker setup and Swarm bootstrap when live consent and provider
contracts allow mutation. It blocks before mutation when readiness, backend
selection, WSL2 capability, or provider profile requirements are not
satisfied. Default `platform reconcile` returns a verified `lxc_native` no-op
boundary after Platform init. Default `platform expose` returns verified
evidence only after the configured manager-profile proxy devices are present
or updated. Default normal flows do not remove direct instance-level
`tsw-proxy-*` devices; those are surfaced as drift and handled only by the
explicit repair workflow.

Continuation requires both parts:

* apply succeeds;
* verify returns `VerificationStatus.verified` evidence.

Platform apply exceptions and explicit failed apply results return
`failed_to_apply`. Artifact preparation failures and deployment pre-upload
failures return `failed_to_prepare`. Verification exceptions, failed
verification results, or non-verified evidence return `failed_to_verify`, while
missing evidence returns `blocked`. Later mutating steps are not run after any
of these outcomes.

When a verification evidence repository is configured, the workflow appends
the typed verification result before making the continuation decision.

=== LXC Proxy Drift Repair Runtime Flow

LXC proxy device ownership is split deliberately:

* desired proxy devices are profile-level devices in
  `docker-swarm-manager`;
* `swarm-manager` receives that manager-only profile in addition to the common
  Docker Swarm profile;
* worker nodes keep only worker-safe profiles;
* direct instance-level `tsw-proxy-*` devices are drift, not expected runtime
  state.

`platform repair-lxc-proxy-drift` is a live-consent-gated mutating workflow.
It does not run during `setup run`, `install.sh`, `platform init`,
`platform reconcile`, `platform expose`, `platform reset`, or
`platform destroy`. The infrastructure adapter reads direct manager instance
devices, filters to project-named `tsw-proxy-*` proxy devices, verifies the
equivalent profile device through the profile-device port, and removes only
that verified direct device. Unknown project proxy names, unrelated devices,
non-proxy devices, missing profile devices, drifted profile devices, and
lookup failures are refused or blocked with summary evidence. Verification
evidence records counts and safe device names, not raw command output.

The non-mutating `platform verify` workflow also interprets preflight results.
Failed preflight checks now return `failed_to_verify` with summary-only
verification evidence instead of reporting a completed verify workflow.
Static preflight does not claim live provider readiness. The default
`lxc_native` path evaluates LXD/Incus readiness, WSL2 capability, backend
candidate selection, and Docker-in-container profile requirements through the
platform init provider guard before mutation.

=== Autonomous Setup Runtime Flow

Autonomous setup now has one canonical workflow-level CLI contract:
`setup run`. The command orchestrates the existing in-process boundaries:

. parse setup intent and selected profile;
. refuse mutating execution before service construction when live consent is
  missing;
. pass the accepted consent object into setup preflight;
. coordinate Platform apply/verify phases through composed workflow objects;
. coordinate Artifacts prepare/verify phases through composed workflow
  objects;
. coordinate Deployment apply/verify phases through composed workflow objects;
. stop later phases after the first refused, blocked, resource-gated,
  failed-to-apply, failed-to-prepare, or failed-to-verify result;
. serialize only known safe workflow result contracts or safe summary
  mappings.

Current implementation status remains fail-closed below the orchestrator. The
CLI command and phase sequencing are implemented, but the repository still
does not claim a verified end-to-end live installation of all runnable
services. Missing setup consent, missing command-backed verification, missing
artifact/registry live steps, missing external Swarm inputs, and missing
observed service readiness must be reported as `refused`, `blocked`,
`resource_gated`, `failed_to_apply`, `failed_to_prepare`, or
`failed_to_verify`, never as success.

The active workflow
`documentation/workflow/workflow.md` implements the fresh-install entry point
for `install.sh`: the wrapper requires the destructive reset confirmation,
runs `platform reset` for the configured managed Tiny Swarm World local
system, stops when reset fails, and only then runs the selected setup profile.
The selected setup path includes artifact preparation, service-stack
deployments, service readiness checks, and service-access dashboard/index
assets. Existing non-destructive update and reconcile mechanisms remain
available through their own workflow commands.

Repository-level wiring and static/mocked evidence cover the fresh-install
sequence and service-access assets, but the repository still does not claim a
verified end-to-end live installation of all runnable services. Live
LXD/Incus, Docker Swarm, registry, Portainer, Nexus, service-access, and
Vaultwarden evidence must be collected during an explicit operator live run
before live success is claimed.

The provider selected by setup defaults to `lxc_native` through LXD or Incus.
Backend selection first honors an explicit `--lxc-backend` override, then
`backend_selection.preferred_backend`, then the ordered
`backend_selection.candidates` list in
`infra/config/node-providers/provider_config.yaml`. Unsupported provider values
fail closed and are not used as silent fallbacks after LXD/Incus readiness
fails.

Optional live smoke validation remains a separate operator action and is not
part of default architecture tests or the default quality gate.
