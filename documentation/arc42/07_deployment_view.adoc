= Deployment View

Deployment strategy and distribution.

== Local Runtime State

Tiny Swarm World remains a single local Python automation application for this
workflow slice. Desired inventory belongs to committed configuration and may be
stored under `infra/config`.

Observed inventory and verification evidence are deployment-local runtime
artifacts. They are stored under the ignored `.tiny-swarm-world/` directory in
the repository root. That directory is not a deployment input, is not part of
the default quality gate outputs, and must not be committed or moved under
`infra/config`.

== Deployment Contracts

The in-process CLI now has explicit artifact and deployment workflow contracts.
`deployment apply` is live-consent gated and now has post-bootstrap
Portainer-managed stack contracts. First-time Portainer and Nexus bootstrap
remain separate bootstrap steps, and service readiness still requires observed
Swarm service evidence. `deployment verify` uses observed-state readiness
checks and must not report success without collected service state.

This keeps the current local deployment model deterministic in tests: the
Python composition root can construct deployment workflow objects without
contacting Portainer, contacting Nexus, building images, pushing registries, or
changing Docker Swarm state. Tests use fakes for Portainer and compose
repositories. Artifact contract tests use fake Nexus clients and patched Docker
CLI calls only; they do not contact Nexus, build images, log in, or push.
Base service stack contracts cover Portainer, Nexus, Jenkins, RabbitMQ,
SonarQube, and Swagger/NGINX. The full guided setup selects the service-access
profile by default and adds the `service-access` stack. The
Portainer-managed plan excludes Portainer itself to avoid a bootstrap cycle
and can exclude bootstrap-owned Nexus. Stack apply checks record Portainer
stack-registration context only; service readiness is verified separately
through observed Swarm service evidence. The service-access deployment path
also has a pre-apply external Swarm input check for the Vaultwarden
administrator-token Swarm secret reference. Missing input blocks deployment
before Portainer stack create/update runs, and verification evidence records
only neutral presence/source classifications.

== Service Access And Vaultwarden Routing Baseline

The service-access dashboard and Vaultwarden capability is implemented as
repository assets and setup-selected service contracts. The repository now
contains the service-access compose stack, image-packaged dashboard and NGINX
assets, service-stack contracts, setup-manifest requirements and static tests.
The current central `http://localhost` route requires a fresh
provider-specific deployment before it is claimed as live-verified.

The accepted routing baseline is `SERVICE_ACCESS_CENTRAL_NGINX`:

* the service-access stack is the accepted owner of published port `80` and is
  intended to serve `http://localhost` as the local management overview after
  verified provider-specific deployment;
* the service-access stack owns its own NGINX ingress instead of modifying the
  Swagger-owned NGINX service;
* stable central paths such as `/jenkins`, `/nexus`, `/portainer`,
  `/rabbitmq`, `/sonarqube`, `/swagger` and `/vaultwarden` are managed by
  service-access NGINX;
* the Swagger/NGINX stack uses published port `8084`;
* the Vaultwarden HTTPS route uses published port `443`, while `8086` remains a local HTTP compatibility route;
* Traefik remains deferred until a later ADR and tests add a provider, labels,
  dashboard exposure policy, network model and rollback.

Portainer may manage service-access stack create/update only after Portainer
is reachable. First-time Portainer bootstrap remains outside the
Portainer-managed upload path.

For the Portainer-managed path, service-access dashboard and NGINX assets are
image-packaged under `infra/compose/service-access/**`. Bind-mounted
repository files are not an accepted Portainer path unless a later slice adds
a tested asset-preparation port or adapter and documents when it runs.

The Vaultwarden administrator-token secret name is propagated through an
allowlisted stack environment variable for both Portainer-managed and direct
Swarm stack deployment paths. The literal Swarm secret name is a deployment
input and must not be copied into `VerificationResult` evidence.

== Planned Traefik HTTPS Ingress Migration

The accepted migration direction in
`documentation/architecture/adr-traefik-https-ingress-existing-ca.adoc` is to
move toward centralized local HTTPS ingress through Traefik using an existing
operator-provided CA. This is accepted as workflow direction, not yet verified
live runtime behavior.

The planned ingress deployment keeps Tiny Swarm World Docker Swarm-first and
LXC-native. Traefik belongs to the Deployment responsibility boundary. It is
not a new product microservice, not a Kubernetes ingress controller, and not a
browser frontend project.

The migration must preserve these deployment constraints:

* public LXC exposure for ingress is limited to ports `80` and `443`;
* Traefik uses `web` on `:80` and `websecure` on `:443`;
* HTTP redirects to HTTPS;
* `exposedByDefault=false` is required;
* `--api.insecure=true` is forbidden;
* existing CA and certificate private material are operator-owned secrets and
  must not be committed or persisted in evidence;
* hostname resolution through DNS or hosts fallback requires explicit
  verification, and any host mutation requires consent and rollback;
* live browser and TLS checks remain opt-in and write only redacted local
  evidence under ignored `.tiny-swarm-world/evidence/**` paths.

The current service-access NGINX routing baseline remains the comparison and
rollback point until implementation slices verify Traefik routing, LXC public
port reconciliation, certificate validation and browser-visible HTTPS
behavior.

The repository now includes static Traefik Swarm ingress configuration under
`infra/config/compose/traefik/docker-compose.yml` plus the dynamic TLS summary
configuration under `infra/compose/traefik/dynamic/tls.yml`. The Traefik stack
uses Swarm discovery with `exposedByDefault=false`, `web` on `:80`,
`websecure` on `:443`, HTTP-to-HTTPS redirect, external certificate secrets,
and no insecure API flag.

The Jenkins, Nexus, Portainer, SonarQube, and Infisical stack definitions now
carry Traefik Swarm route labels and join the external `traefik_ingress`
network. This is committed desired runtime configuration, not live deployment
evidence. Live route success, LXC public port convergence, hostname
resolution, and browser-visible HTTPS behavior remain pending the later
workflow slices.

== Autonomous Setup Deployment Contract

The canonical `setup run` command is the workflow-level setup orchestrator for
a local Linux/WSL Docker Swarm-first deployment. It is mutating,
non-destructive, and live-consent gated. Full runnable success is reached only
after the selected profile's host prerequisites, platform state, artifact
registry behavior, service stacks, and readiness checks are verified.

The platform init workflow also owns the live-readiness pre-apply guard. Direct
`platform init --live` and setup-driven `platform init` therefore use the same
application workflow boundary before provider node creation, Docker
installation, or Swarm initialization can begin. The default guard is
read-only and classifies LXD/Incus runtime readiness, WSL2 capability,
backend ambiguity, and provider profile requirements; it does not repair host
services, install packages, change groups, create profiles, or create nodes
without accepted live consent.

The provider deployment target is now `lxc_native` through LXD or Incus by
default. The committed provider configuration in
`infra/config/node-providers/provider_config.yaml` declares Incus/LXD backend
candidates, Docker Swarm node names, Docker-in-container profile risk labels,
the common `docker-swarm` profile, the manager-only
`docker-swarm-manager` proxy profile, and verification metadata. The manager
node expects both profiles. Worker nodes expect only worker-safe profiles and
must not receive the manager proxy profile. Deployment evidence must keep
native Linux LXD/Incus and WSL2 LXD/Incus results separate.

For the LXC-native target, deployment must block before mutation when LXD or
Incus readiness is absent, backend selection is ambiguous, WSL2 capability is
missing, or Docker-in-container profile requirements are not satisfied. Host
package installation, provider daemon repair, bridge or firewall mutation,
group membership repair, and privileged container profile changes are live
host mutations and are not deployment defaults in this slice.

Published service port exposure is deployed through manager-profile LXC proxy
devices. Direct instance-level `tsw-proxy-*` devices on `swarm-manager` are
deployment drift and remain unsafe during normal lifecycle checks. The
operator repair surface is the explicit `platform repair-lxc-proxy-drift`
workflow, which removes only stale direct project proxy devices whose
equivalent manager-profile device is verified.

Service-stack contracts carry localhost endpoint defaults so operator-facing
routes are explicit and test-backed. These endpoints are configuration hints
for WSL/Linux local access and do not claim Swarm service health. Localhost
forwarding failures are diagnosed separately from deployment readiness, and
operators may use current Swarm node IPs only as manual diagnostic fallbacks.

Hosts that do not satisfy the full-run resource contract remain
resource-gated. A resource-gated result must identify omitted services and must
not be reported as a full deployment pass.

Default verification remains mocked or static. Optional live smoke validation
requires explicit operator approval, live consent, and a disposable or
recoverable local target environment. Live smoke evidence must be written only
as redacted local state and must remain separate from the default quality gate.
