== Architectural Decisions

This section indexes active architecture decisions and implementation decisions
from the safety-first workflow. Standalone ADR files keep their own history and
status; arc42 summarizes the current consequences.

=== ADR Location Convention

Standalone ADR files currently live under `documentation/architecture/` with
the naming pattern `adr-<decision-slug>.adoc`.

`documentation/arc42/09_architecture_decisions.adoc` is the active index for
those decisions. Existing ADR files must not be silently moved, renumbered or
renamed as part of workflow execution.

The repository does not currently use `documentation/adr/**`. Introducing that
path requires a separate convention decision, including
`documentation/adr/README.md`, reference migration, and history-preserving
links from existing ADR files.

=== Platform, Artifacts, Deployment, And Shared Responsibilities

ADR: `documentation/architecture/adr-separate-platform-artifacts-deployment.adoc`

Status: accepted as responsibility direction, partially implemented.

Consequences:

* `ApplicationServices` aggregates `PlatformServices`, `ArtifactServices`, and
  `DeploymentServices`;
* Platform workflows own Multipass, netplan, Docker install, Swarm, preflight,
  and VM/IP behavior;
* the Deployment namespace owns `EnsureNexusStack`, and deployment workflow
  composition owns explicit blocked contracts until live Portainer behavior is
  implemented;
* Artifacts owns explicit blocked workflow contracts until artifact registry
  and publishing behavior is implemented;
* these bundles are not independently deployable microservices.

=== Autonomous Setup Safety Contract

ADR: `documentation/architecture/adr-autonomous-setup-safety.adoc`

Status: accepted as setup-safety contract, implemented for the canonical
fail-closed setup CLI orchestration contract.

Consequences:

* autonomous setup is a consent-gated orchestration over existing in-process
  Platform, Artifacts, Deployment, Shared, and Console/status UI boundaries;
* mutating setup must preserve `--live` and the short interactive
  live-infrastructure confirmation until a later ADR changes that contract;
* `setup run` passes accepted live consent into setup preflight, sequences only
  non-destructive phases, and stops later phases after refused, blocked,
  resource-gated, failed-to-apply, or failed-to-verify outcomes;
* missing consent, missing verification, resource-gated outcomes, and failed
  phases must be reported explicitly and fail closed;
* automatic host package installation and non-interactive live setup remain out
  of scope until separate ADRs accept those contracts;
* direct scripts are not the canonical setup path unless later wrapped behind
  ports, adapters, tests, verification, and consent controls;
* setup evidence remains redacted local state and must not persist raw command
  payloads, secrets, Swarm tokens, host-specific addresses, or environment
  payloads.

=== Service Access Dashboard And Vaultwarden Baseline

ADR: `documentation/architecture/adr-service-access-dashboard-vaultwarden.adoc`

Status: accepted as requirement and security baseline; implemented as the full
guided setup service-access profile with residual operational hardening.

Consequences:

* the service-access capability belongs to the Deployment responsibility
  boundary as a Docker Swarm service stack;
* Vaultwarden is the only approved password reveal and copy surface;
* the service-access dashboard may show service names, access routes,
  reachability states, credential labels and Vaultwarden item references, but
  must not display, cache, export, log or persist password values;
* the full guided setup includes service-access through
  `ServiceStackProfile.SERVICE_ACCESS`; the older base service set remains
  available through `--service-profile default`;
* NGINX is the baseline routing direction, with dedicated local ports `8085`
  and `8086` for the service-access stack;
* Traefik, TLS automation and wider-than-local exposure require later ADR
  coverage;
* Portainer may manage post-bootstrap stack create/update only after
  Portainer is reachable and must not become its own bootstrap dependency;
* live smoke evidence verified Swarm node IP reachability on 2026-05-25, while
  local `localhost` forwarding and Vaultwarden operational hardening remain
  follow-up work.

=== Workflow Taxonomy And Destructive Operation Policy

Decision status: implemented in the safety-first workflow.

`init`, `reconcile`, `verify`, `reset`, and `destroy` are explicit platform
workflow semantics. Normal `init` and `reconcile` are non-destructive.
`reset` and `destroy` require exact confirmation and remain blocked until
retention/teardown semantics are implemented.

=== Workflow-Level CLI

Decision status: implemented for platform workflows, blocked artifact and
deployment contracts, and the fail-closed `setup run` orchestration contract.

The Python entry point lists workflow-level commands, refuses unsupported
workflow names, and applies live-consent checks before constructing services
for mutating workflows. Artifact and deployment workflow names dispatch to
blocked workflow contracts until live behavior lands. `setup run` is mutating
and non-destructive; it is listed by the CLI, refuses missing live consent
before setup service construction, and delegates phase sequencing to
`tiny_swarm_world.application.services.setup`.

=== Typed Command Catalog

Decision status: implemented for existing command YAML files.

Command YAML under `infra/config/**/command_*.yaml` is product configuration.
Entries include command identity, intent, execution mode, safety class, scope,
allowed workflows, parameters, effects, and verification specification.
Application command building receives the active workflow identifier and
rejects commands outside their allow-list.

=== Inventory State Storage Ownership

Decision status: implemented in Slice 04.

Tiny Swarm World separates committed desired inventory from ignored local
observed inventory and verification evidence.

Consequences:

* desired inventory remains a product configuration concern and may live under
  `infra/config`;
* observed inventory and verification evidence live only under
  `.tiny-swarm-world/`;
* the domain owns typed state and evidence concepts without filesystem, YAML,
  HTTP, command-runner, Docker, logging, or infrastructure imports;
* infrastructure adapters own YAML/JSON parsing, local path resolution, and
  persistence hardening;
* local evidence stores summaries and rejects raw command payload fields.

=== Verify After Apply

Decision status: implemented for mutating platform workflows.

Mutating workflow continuation requires apply success and verified evidence.
Apply failures return `failed_to_apply`, verification failures return
`failed_to_verify`, and missing verification paths or missing evidence return
`blocked`.

=== ADR Backlog

The typed command catalog, inventory/evidence ownership, workflow-level CLI,
and verify-after-apply decisions are currently recorded in this arc42 index and
the active workflow. Dedicated standalone ADRs can be added later if the
project decides these policies need separate long-lived records.
