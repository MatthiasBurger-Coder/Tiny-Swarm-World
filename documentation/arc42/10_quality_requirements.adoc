== Quality Requirements

Critical quality aspects and non-functional requirements.

=== Command Catalog Safety

The command catalog must fail closed before command execution:

* every command YAML entry has a stable `id` and complete typed metadata;
* duplicate command IDs or duplicate execution indexes are rejected;
* destructive command strings without `safety_class: destructive` are rejected;
* `runtime_change` command effects are rejected when classified as `safe_read`;
* `credential_output` command effects are rejected unless the command is
  `credential_mutation` with redacted, non-raw evidence policy;
* mutating commands without a verification specification are rejected;
* command usage is rejected when the active workflow is not listed in
  `allowed_workflows`.

The default quality gate must validate this contract without executing live
infrastructure commands. Regression coverage includes domain model validation,
YAML repository validation, workflow allow-list enforcement, and static checks
that destructive command patterns are reset/destroy only. Desired inventory
tests must load the committed host-neutral baseline and reject host-specific or
secret-bearing fields.

=== Verify After Apply

Mutating workflows must not silently continue after applying a change.

Quality scenarios:

* a mutating workflow step without a verification path returns `blocked` before
  the apply step runs;
* an apply exception or failed apply result returns `failed_to_apply` and stops
  later steps;
* a verification exception or failed verification result returns
  `failed_to_verify` and stops later steps;
* missing verification evidence returns `blocked`;
* verified evidence allows the next mutating step to run.

Tests use synthetic workflow steps and mocked repositories. They do not execute
Multipass, Docker Swarm, compose deployments, netplan, socat, or service
bootstrap commands.

=== Autonomous Setup Safety

Autonomous setup quality gates must prove fail-closed behavior without live
infrastructure side effects by default.

Quality scenarios:

* mutating setup without all required live-consent signals is refused before
  Multipass, Docker Swarm, netplan, socat, compose, Portainer, Nexus, service
  bootstrap, image build, image push, or stack upload commands can run;
* preflight reports missing host prerequisites, resource limits, credentials,
  port conflicts, or configuration blockers before mutation;
* accepted live preflight distinguishes Multipass executable presence from
  daemon, socket, permission, and driver readiness before VM mutation;
* Multipass readiness diagnostics expose safe classifications and return codes,
  not raw stdout, stderr, command payloads, socket paths, user names, or
  host-specific paths;
* missing verification contracts return `blocked`, not success;
* apply failures return `failed_to_apply` and stop later setup phases;
* verification failures return `failed_to_verify` and stop later setup phases;
* resource-gated profiles are distinct from full runnable success;
* evidence persistence rejects raw command strings, stdout, stderr,
  environment payloads, tokens, passwords, Swarm join tokens, host-specific
  addresses, and user-specific absolute paths;
* default quality gates and architecture tests do not execute live
  infrastructure commands;
* optional live smoke validation is documented and invoked separately from the
  default quality gate.

=== Service Access And Credential Display Safety

The service-access dashboard and Vaultwarden stack must preserve credential
and reachability boundaries without live infrastructure side effects in the
default quality gate.

Quality scenarios:

* the base stack profile excludes `service-access`;
* the full guided setup profile includes the `service-access` stack with
  required services `service-access-dashboard`, `vaultwarden`, and
  `service-access-nginx`;
* the default setup CLI plan lists the service-access landing page and
  Vaultwarden ports, while `--service-profile default` can omit the management
  stack intentionally;
* committed service-access YAML loads through the compose repository and
  publishes only the service-access NGINX ports `80` and `8086`;
* Vaultwarden receives its administrator token through an external Swarm
  secret name and no committed token value or static password default exists;
* service-access dashboard and NGINX images are part of the artifact image
  publication contracts for full setup;
* dashboard and NGINX assets are image-packaged and do not depend on
  repository bind mounts for the Portainer-managed path;
* dashboard links use central service-access route paths and do not include
  credential-bearing query strings, fragments, or userinfo URLs;
* dashboard `pwd` cells show Vaultwarden item references instead of raw
  password values;
* no React, Vite, TSX, JSX, package-manager lockfile, host-side live helper,
  Docker build script, Portainer upload script, token, password, API key, or
  credential-bearing URL is introduced in service-access assets.

Service-access readiness must remain unknown or failed-to-verify until
observed Swarm service state or a later HTTP evidence source verifies the
selected service stack.
