== Quality Requirements

Critical quality aspects and non-functional requirements.

=== Command Catalog Safety

The command catalog must fail closed before command execution:

* every command YAML entry has a stable `id` and complete typed metadata;
* duplicate command IDs or duplicate execution indexes are rejected;
* destructive command strings without `safety_class: destructive` are rejected;
* `runtime_change` command effects are rejected when classified as `safe_read`;
* `credential_output` command effects are rejected unless the command is
  `credential_mutation` with redacted, non-raw evidence policy;
* mutating commands without a verification specification are rejected;
* command usage is rejected when the active workflow is not listed in
  `allowed_workflows`.

The default quality gate must validate this contract without executing live
infrastructure commands. Regression coverage includes domain model validation,
YAML repository validation, workflow allow-list enforcement, and static checks
that destructive command patterns are reset/destroy only. Desired inventory
tests must load the committed host-neutral baseline and reject host-specific or
secret-bearing fields.

=== Verify After Apply

Mutating workflows must not silently continue after applying a change.

Quality scenarios:

* a mutating workflow step without a verification path returns `blocked` before
  the apply step runs;
* an apply exception or failed apply result returns `failed_to_apply` and stops
  later steps;
* a verification exception or failed verification result returns
  `failed_to_verify` and stops later steps;
* missing verification evidence returns `blocked`;
* verified evidence allows the next mutating step to run.

Tests use synthetic workflow steps and mocked repositories. They do not execute
Docker Swarm, compose deployments, netplan, socat, or service
bootstrap commands.

=== Autonomous Setup Safety

Autonomous setup quality gates must prove fail-closed behavior without live
infrastructure side effects by default.

Quality scenarios:

* mutating setup without all required live-consent signals is refused before
  Incus, LXC container lifecycle, Docker Swarm, netplan,
  socat, compose, Portainer, Nexus, service bootstrap, image build, image
  push, or stack upload commands can run;
* preflight reports missing host prerequisites, resource limits, credentials,
  port conflicts, or configuration blockers before mutation;
* accepted live preflight distinguishes Incus executable presence, daemon
  access, backend candidate selection, WSL2 capability, and Docker-in-container
  profile requirements before LXC-native node mutation;
* direct `platform init --live` and setup-driven platform init both stop before
  mutating steps when the shared live preflight guard fails;
* provider readiness diagnostics expose safe classifications and return codes,
  not raw stdout, stderr, command payloads, socket paths, user names, or
  host-specific paths;
* default `lxc_native` setup and platform paths do not construct removed
  provider clients;
* missing provider-native artifact or deployment prerequisites return blocked
  or failed workflow results instead of silently falling back to another
  provider or claiming success;
* missing verification contracts return `blocked`, not success;
* apply failures return `failed_to_apply` and stop later setup phases;
* verification failures return `failed_to_verify` and stop later setup phases;
* resource-gated profiles are distinct from full runnable success;
* evidence persistence rejects raw command strings, stdout, stderr,
  environment payloads, tokens, passwords, Swarm join tokens, host-specific
  addresses, and user-specific absolute paths;
* default quality gates and architecture tests do not execute live
  infrastructure commands;
* optional live smoke validation is documented and invoked separately from the
  default quality gate.

=== Registry And Validation Plans

Quality scenarios:

* installation phases sort deterministically by dependency, order, and phase
  identifier;
* dependency cycles, unknown dependencies, missing required phases, missing
  required services, and missing required workflow phases fail closed;
* service-stack contracts map to declared installation phases and port-registry
  IDs;
* compose published ports are classified as registry-backed, Traefik ingress,
  compatibility, or deferred;
* health-check registry entries match service registry and service-stack
  contracts;
* validation-plan targets cover the required health-check targets;
* missing required observed evidence returns `failed_to_verify`;
* verified static stack-registration evidence does not satisfy service
  readiness.

=== Installation Progress And Cross-Cutting Method Trace Logging

Installation progress and logging are implemented through two correlated
observability paths:

* high-level workflow progress for operator-facing setup phase and platform
  step status;
* cross-cutting method trace logging for the covered installation runtime path,
  including method entry, normal exit, and exception exit.

ADR `documentation/architecture/adr-cross-cutting-method-trace-logging.adoc`
governs the method trace direction. The trace implementation must not spread
concrete terminal UI or logger configuration dependencies through setup,
platform, domain, or application services.

Quality scenarios:

* setup progress emits start and terminal events for refused, blocked,
  completed and failed outcomes;
* every setup phase emits start and terminal events, and downstream phases
  that are skipped after a terminal non-success state are reported as
  `not_run`;
* platform progress emits pre-apply guard, mutating step start, apply result,
  direct verification, verify step, blocked, failed apply, failed verify and
  completed events;
* refused, blocked, failed, failed-to-apply and failed-to-verify states are
  not overwritten by later success-like aggregate updates;
* every method in the documented installation trace scope emits `entered`,
  `returned`, and `raised` lifecycle events, or has an explicit tested
  exemption;
* exception paths emit a safe failure trace before propagation or conversion to
  a workflow result;
* asynchronous and nested method traces carry correlation and parent-span
  metadata sufficient to reconstruct program flow;
* progress events and logs carry only safe summaries and recovery hints;
* trace events, progress events and logs do not contain raw arguments, return
  payloads, raw exception messages, stack traces, raw command strings, stdout,
  stderr, tokens, passwords, environment payloads, host-specific paths or local
  IP addresses;
* tests or static checks fail when a covered installation runtime method lacks
  entry, returned, or raised trace coverage;
* default tests and quality gates use mocks or static inspection and do not
  execute live Incus, LXC, Docker Swarm, compose, Portainer,
  Nexus, Jenkins, Apache Pulsar, SonarQube or service bootstrap commands.

=== Service Access And Credential Display Safety

The service-access dashboard and stack metadata must preserve credential and
reachability boundaries without live infrastructure side effects in the
default quality gate.

Quality scenarios:

* the base stack profile excludes `service-access`;
* the full guided setup profile includes the `service-access` stack with
  required services `service-access-dashboard` and `service-access-nginx`;
* the default setup CLI plan lists compose service names and published ports
  parsed from the selected stacks, while `--service-profile default` can omit
  the management stack intentionally;
* committed service-access YAML loads through the compose repository and
  publishes only the service-access NGINX ports `10000` and `8086`;
* Portainer and direct Swarm deployment paths propagate only allowlisted stack
  environment values, and verification evidence records no literal token,
  secret, password or credential names;
* service-access dashboard and NGINX images are part of the artifact image
  publication contracts for full setup;
* dashboard and NGINX assets are image-packaged and do not depend on
  repository bind mounts for the Portainer-managed path;
* dashboard links use central service-access route paths and do not include
  credential-bearing query strings, fragments, or userinfo URLs;
* dashboard `pwd` cells show secret item references instead of raw password
  values;
* no React, Vite, TSX, JSX, package-manager lockfile, host-side live helper,
  Docker build script, Portainer upload script, token, password, API key, or
  credential-bearing URL is introduced in service-access assets.

Service-access readiness must remain unknown or failed-to-verify until
observed Swarm service state or a later HTTP evidence source verifies the
selected service stack.

=== LXC Proxy Profile And Drift Safety

LXC-native proxy exposure must preserve strict drift detection while allowing
an explicit operator repair path.

Quality scenarios:

* the manager node expects the common Docker Swarm profile plus the
  manager-only `docker-swarm-manager` profile;
* worker nodes do not receive the manager proxy profile;
* `platform expose` creates or updates profile-level proxy devices and does
  not create direct instance-level `tsw-proxy-*` devices;
* direct instance-level `tsw-proxy-*` devices remain
  `unsafe_instance_devices` during normal node lifecycle checks;
* normal `platform init`, `platform reconcile`, `platform expose`,
  `platform reset`, `install.sh`, and `setup run` do not silently remove stale
  direct proxy devices;
* `platform repair-lxc-proxy-drift` requires live consent before mutation;
* repair removes only direct `tsw-proxy-*` proxy devices that have an
  equivalent verified manager-profile device;
* repair refuses unrelated devices, unknown project proxy names, and direct
  project proxy devices whose equivalent manager-profile state is missing,
  drifted, or unobservable;
* repair evidence records summary counts and safe device names without raw
  command strings, stdout, or stderr.

=== Portainer Admin Initialization Rejection Gate

Deployment quality gates must prove that Portainer admin bootstrap distinguishes
transient readiness failures from explicit Portainer rejection.

Quality scenarios:

* transient admin initialization failures are retried until authentication
  succeeds or the attempt budget is exhausted;
* a typed `PortainerAdminInitializationRejected` failure fails fast when the
  requested credentials still cannot authenticate, and is not retried as a
  readiness delay;
* LXC-native Portainer admin adapters translate
  rejected HTTP initialization responses into the typed application-port
  exception when the requested credentials cannot authenticate;
* HTTP rejection after successful authentication is treated as already usable
  admin access and does not raise the typed rejection;
* tests use mocked clients or request sessions and do not execute live
  Incus, LXC, Docker Swarm, compose, Portainer, Nexus, Jenkins,
  Apache Pulsar, SonarQube or service bootstrap commands.
