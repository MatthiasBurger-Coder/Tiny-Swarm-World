= Deployment

:sectnums:
:toc:

== Purpose

Tiny Swarm World deploys local infrastructure services into a Docker Swarm
environment whose default node provider is managed LXC through LXD or Incus.
The deployment documentation is intentionally limited to the workflows that
exist in this repository today.

This project is developed and operated from WSL or Linux shells. Windows-specific shell commands are not the default execution model.

== Deployment model

The repository separates deployment work into three areas:

* `src/tiny_swarm_world/__main__.py` contains the Python entry point for infrastructure orchestration.
* `infra/compose` contains image build-context assets for workflow-managed deployment.

The active Python entry point exposes workflow-level commands. Running without
arguments does not execute infrastructure commands, and `--list-workflows`
returns the declared workflow list without building application services.
Static preflight is available through `--preflight` for the default
`lxc_native` provider.

Implemented platform workflows are `platform init`, `platform reconcile`,
`platform expose`, `platform repair-lxc-proxy-drift`, `platform verify`,
`platform reset`, and `platform destroy`. Mutating platform workflows require
live consent before service construction. Reset and destroy also require exact
confirmation and currently block after confirmation until retention and
teardown semantics are implemented. Default platform init uses
the `lxc_native` provider selection path and blocks before mutation when
LXD/Incus readiness or profile requirements are not satisfied. Default
platform expose creates or updates profile-level LXC proxy devices in the
manager-specific `docker-swarm-manager` profile for published setup-manifest
ports. The `swarm-manager` node receives that profile in addition to the common
Docker Swarm profile; workers do not receive it. Direct instance-level
`tsw-proxy-*` devices on `swarm-manager` are drift, not expected deployment
state. The explicit `platform repair-lxc-proxy-drift` workflow can remove stale
direct `tsw-proxy-*` devices only after the equivalent manager-profile device
is verified. Platform reconcile, artifact workflows, and deployment workflows
block at provider-native boundaries until their LXC-native contracts are wired.
Unsupported node-provider values fail closed; they are not remapped to another
provider.

Deployment bootstrap includes Portainer application initialization. After the
Portainer stack and admin credentials are available, the bootstrap workflow
ensures the local Docker endpoint named `local` exists in Portainer before
Portainer-managed application stacks are applied.

The Deployment boundary now contains a tested post-bootstrap
`EnsurePortainerStack` application service and a sanitized Portainer HTTP
adapter contract. The CLI composition root does not wire that contract as a
successful live `deployment apply` path yet, because first-time Portainer
bootstrap and verify-after-apply evidence remain incomplete.

The Deployment boundary also contains a generic `EnsureServiceStack` contract
and stack plans for the default service set plus selected service profiles.
The base service set covers Portainer, Nexus, Jenkins, RabbitMQ, SonarQube,
and Swagger/NGINX. The full guided setup selects the service-access profile by
default, which adds the `service-access` management stack. The
Portainer-managed plan intentionally excludes the Portainer stack itself to
avoid a bootstrap cycle and can exclude bootstrap-owned Nexus. These
contracts can apply a stack through the Portainer port, then record
stack-registration context. They do not claim container health or application
readiness; readiness remains separate observed-state verification.

The canonical autonomous setup command is `setup run`. It is a workflow-level
orchestrator over preflight, platform, artifacts, deployment, and final
verification phases. It refuses missing live consent before setup service
construction and preserves blocked, resource-gated, failed-to-apply, and
failed-to-verify states instead of reporting full deployment success without
observed evidence.

Setup preflight is idempotent for an already running Tiny Swarm World
environment. Required ports pass when they are either free or already serve the
expected local service, reported as `source=existing_expected_service`.
Unknown port owners still fail before live execution.

== Development gate before deployment work

Run the development quality gate before changing deployment-related code:

[source,bash]
----
python3 tools/quality_gate.py quality
----

When a virtual environment is active, `python` can be used instead of `python3`.

The quality gate covers linting, import architecture checks, architecture
tests, type checks, and unittest discovery. It does not start LXD/Incus/LXC
containers, initialize Docker Swarm, or deploy live services.

== Service preparation

Portainer, Nexus, image publication, and stack deployment are owned by the
canonical setup workflow boundaries. On the default `lxc_native` path they
remain blocked until provider-native artifact, deployment, and observed-state
contracts are wired:

[source,bash]
----
PYTHONPATH=src python3 -m tiny_swarm_world setup run --live
----

Former direct preparation scripts and host-side orchestration scripts have
been removed so setup behavior stays behind the workflow-level CLI consent
guard. They are not development quality-gate commands.
The Portainer compose asset under `infra/config/compose/portainer` mounts the
host Docker socket for the Portainer server and agent, and the agent also
mounts Docker volumes. Treat that asset as privileged Docker-control
configuration for intentional local Swarm bootstrap only.
The canonical live-operation surface catalog is
`documentation/system/live-operation-surfaces.adoc`.

== Nexus bootstrap

Nexus setup is owned by the Python setup workflow boundary. The default
`lxc_native` path does not yet claim live Nexus stack creation,
administrator-access establishment, or repository readiness success. Those
steps remain blocked until provider-native artifact and deployment contracts
are wired and verified.

Important environment variables:

[options="header"]
|===
| Variable | Default | Purpose
| `TSW_PORTAINER_URL` | `http://localhost:9000` | Portainer API base URL
| `TSW_PORTAINER_USERNAME` | `admin` | Portainer login user
| `TSW_PORTAINER_ADMIN_PASSWORD` | no default, operator supplied | Portainer login password
| `TSW_PORTAINER_ENDPOINT` | `local` | Portainer endpoint name
| `TSW_NEXUS_STACK_NAME` | `nexus` | Stack name in Portainer
| `TSW_NEXUS_URL` | `http://localhost:8081` | Nexus base URL
| `TSW_NEXUS_ADMIN_USERNAME` | `admin` | Nexus administrator user
| `TSW_NEXUS_ADMIN_PASSWORD` | no default, operator supplied | New Nexus administrator password
| `TSW_NEXUS_INITIAL_PASSWORD_PATH` | `/nexus-data/admin.password` | Initial password path inside the Nexus container
| `TSW_NEXUS_MAX_ATTEMPTS` | `10` | Startup polling attempts
| `TSW_NEXUS_WAIT_SECONDS` | `5` | Delay between polling attempts
| `TSW_LXC_DOCKER_REGISTRY_MIRROR` | no default | Optional Docker registry mirror URL written into managed LXC nodes during Docker installation. Use an address reachable from inside the nodes, for example `http://<lxc-reachable-host-ip>:5001`; localhost values are rejected.
|===

The localhost defaults above are operator targets for the Linux/WSL execution
context. They may depend on WSL forwarding state and must not be treated as
proof that a service is live or healthy.

== Stack deployment

The supported stack deployment boundary is the workflow-level setup command.
Compose files under `infra/config/compose` are read by Python deployment
adapters. On the default `lxc_native` path, provider-native stack deployment
remains blocked until the reviewed setup workflow wires and verifies those
contracts.

== Service-access routing and Vaultwarden

The service-access dashboard and Vaultwarden capability is implemented as
repository assets, setup-selected service contracts, and routing design. It is
not claimed as a live-installed management surface for the default
`lxc_native` path until provider-native deployment and service readiness are
verified. The current repository design makes the service-access NGINX the
intended central local HTTP entry point at `http://localhost`. Swagger/NGINX
moves to its own `8084` published route after a verified deployment.

Implemented repository behavior:

* `infra/config/compose/service-access/docker-compose.yml` defines the
  dashboard, Vaultwarden and service-access NGINX stack.
* `infra/compose/service-access/dashboard/**` and
  `infra/compose/service-access/nginx/**` provide image-packaged static
  assets for the Portainer-managed path.
* The setup CLI uses `ServiceStackProfile.SERVICE_ACCESS` by default so the
  selected setup profile includes the service-access landing page contract.
  Operators can use `--service-profile default` to run only the older base
  service set.
* The setup manifest for the service-access profile records ports `80`, `8086`, and `443`
  plus the `TSW_VAULTWARDEN_ADMIN_TOKEN_SECRET` credential-source name only.
  It does not record a token value.

The accepted routing decision is
`SERVICE_ACCESS_CENTRAL_NGINX`:

* Service Access owns published port `80` and serves the overview at
  `http://localhost`.
* Service Access NGINX manages stable local paths such as `/jenkins`,
  `/nexus`, `/portainer`, `/rabbitmq`, `/sonarqube`, `/swagger` and
  `/vaultwarden`.
* Swagger/NGINX uses published port `8084`.
* The Vaultwarden HTTPS route uses published Swarm port `443`; `8086` remains a local HTTP compatibility route.
* Wider-than-local exposure, Traefik, and public certificate automation require
  later ADR coverage and tests.

Portainer may manage the service-access stack only after Portainer is
reachable. First-time Portainer bootstrap remains outside Portainer-managed
stack upload.

For a Portainer-managed service-access stack, dashboard and NGINX assets are
image-packaged. Do not rely on bind-mounted repository files unless a later
slice adds a tested asset-preparation boundary.

The static dashboard is a management table with `Server`, `URL`, `user` and
`pwd` columns. Service entries use central route paths and Vaultwarden item
references. Password values remain visible only through Vaultwarden's
authenticated reveal/copy flow.

Current live verification note:

* The central `http://localhost` route requires a fresh provider-specific
  service-access stack deployment before it is claimed as live-verified.

== Verification

Useful checks after deployment:

[source,bash]
----
incus list swarm-manager --format json
lxc list swarm-manager --format json
docker node ls
docker stack ls
docker service ls
----

Use either the `incus` or `lxc` command according to the selected backend.

Service-specific web interfaces depend on the published stack ports. Typical local defaults are documented in `documentation/Howto.adoc`.

Optional live smoke validation is separate from the development quality gate
and should be run only on a disposable or recoverable local Swarm target:

[source,bash]
----
PYTHONPATH=src python3 -m tiny_swarm_world setup run --live
----

When prompted, answer `y` only if the operator intentionally accepts local
LXD/Incus/LXC provider state, Docker Swarm,
networking, service bootstrap, image, and stack changes. A live smoke result
is evidence for that local run only and must not be treated as the default
repository quality gate.

== Operational caution

The guided setup workflow can create or modify provider nodes, Docker
configuration, Swarm state, network settings, and service credentials. Review
the live-operation surface catalog and environment values before running it
against an existing local environment.

Removed legacy helper areas remain documented as unsupported historical surfaces
until a dedicated cleanup slice verifies replacement parity. Do not add,
restore, or promote any direct live-operation script without updating
`documentation/system/live-operation-surfaces.adoc`, tests, and rollback notes.
