= Deployment

:sectnums:
:toc:

== Purpose

Tiny Swarm World deploys local infrastructure services into a Docker Swarm environment that is usually backed by Multipass virtual machines. The deployment documentation is intentionally limited to the workflows that exist in this repository today.

This project is developed and operated from WSL or Linux shells. Windows-specific shell commands are not the default execution model.

== Deployment model

The repository separates deployment work into three areas:

* `src/tiny_swarm_world/__main__.py` contains the Python entry point for infrastructure orchestration.
* `infra/prepare` contains retired notes for former direct service preparation helpers.
* `infra/compose` contains image build-context assets for workflow-managed deployment.

The active Python entry point exposes workflow-level commands. Running without
arguments does not execute infrastructure commands, and `--list-workflows`
returns the declared workflow list without building application services.
Static preflight is available through `--preflight`.

Implemented platform workflows are `platform init`, `platform reconcile`,
`platform verify`, `platform reset`, and `platform destroy`. Mutating platform
workflows require live consent before service construction. Reset and destroy
also require exact confirmation and currently block after confirmation until
retention and teardown semantics are implemented. Artifact and deployment
workflow names dispatch to explicit fail-closed workflow contracts until live
Nexus, registry, Portainer, image publication, and observed-state verification
contracts are wired in composition. The artifact workflow contracts can now
run configured Nexus repository checks with typed verification evidence, but
the default CLI composition still keeps artifact live behavior blocked.

The Deployment boundary now contains a tested post-bootstrap
`EnsurePortainerStack` application service and a sanitized Portainer HTTP
adapter contract. The CLI composition root does not wire that contract as a
successful live `deployment apply` path yet, because first-time Portainer
bootstrap and verify-after-apply evidence remain incomplete.

The Deployment boundary also contains a generic `EnsureServiceStack` contract
and a default stack plan for Portainer, Nexus, Jenkins, RabbitMQ, SonarQube,
and Swagger/NGINX. The Portainer-managed plan intentionally excludes the
Portainer stack itself to avoid a bootstrap cycle. These contracts can apply a
stack through the Portainer port, then record stack-registration context while
returning blocked readiness until required services are observed. They do not
claim container health or application readiness; those checks remain future
observed-state contracts.

The canonical autonomous setup command is `setup run`. It is a workflow-level
orchestrator over preflight, platform, artifacts, deployment, and final
verification phases. It refuses missing live consent before setup service
construction and preserves blocked, resource-gated, failed-to-apply, and
failed-to-verify states instead of reporting full deployment success without
observed evidence.

== Development gate before deployment work

Run the development quality gate before changing deployment-related code:

[source,bash]
----
python3 tools/quality_gate.py quality
----

When a virtual environment is active, `python` can be used instead of `python3`.

The quality gate covers linting, import architecture checks, architecture
tests, type checks, and unittest discovery. It does not start Multipass VMs,
initialize Docker Swarm, or deploy live services.

== Service preparation

Portainer, Nexus, image publication, and stack deployment are started through
the canonical setup workflow:

[source,bash]
----
PYTHONPATH=src python3 -m tiny_swarm_world setup run --live
----

Former direct scripts under `infra/prepare` and host-side orchestration scripts
under `infra/compose` have been retired so setup behavior stays behind the
workflow-level CLI consent guard. Direct scripts under `infra/swarm` remain
legacy live-operation surfaces and are not development quality-gate commands.
The Portainer compose asset under `infra/config/compose/portainer` mounts the
host Docker socket for the Portainer server and agent, and the agent also
mounts Docker volumes. Treat that asset as privileged Docker-control
configuration for intentional local Swarm bootstrap only.
The canonical live-operation surface catalog is
`documentation/system/live-operation-surfaces.adoc`.

== Nexus bootstrap

Nexus setup is owned by the Python setup workflow. The workflow creates or
updates the Nexus stack through the deployment boundary, establishes
administrator access, and applies artifact repository readiness contracts.

Important environment variables:

[options="header"]
|===
| Variable | Default | Purpose
| `TSW_PORTAINER_URL` | `http://localhost:9000` | Portainer API base URL
| `TSW_PORTAINER_USERNAME` | `admin` | Portainer login user
| `TSW_PORTAINER_PASSWORD` | local compatibility default, override required | Portainer login password
| `TSW_PORTAINER_ENDPOINT` | `local` | Portainer endpoint name
| `TSW_NEXUS_STACK_NAME` | `nexus` | Stack name in Portainer
| `TSW_NEXUS_URL` | `http://localhost:8081` | Nexus base URL
| `TSW_NEXUS_ADMIN_USERNAME` | `admin` | Nexus administrator user
| `TSW_NEXUS_ADMIN_PASSWORD` | local compatibility default, override required | New Nexus administrator password
| `TSW_NEXUS_INITIAL_PASSWORD_PATH` | `/nexus-data/admin.password` | Initial password path inside the Nexus container
| `TSW_NEXUS_MAX_ATTEMPTS` | `10` | Startup polling attempts
| `TSW_NEXUS_WAIT_SECONDS` | `5` | Delay between polling attempts
|===

== Stack deployment

The supported stack deployment path is the workflow-level setup command. Compose
files under `infra/config/compose` are read by Python deployment adapters and
deployed through the reviewed setup workflow.

== Planned service-access routing

The service-access dashboard and Vaultwarden capability is currently a planned
deployment baseline, not an implemented stack.

The accepted routing decision for future implementation is
`NGINX_FIRST_DEDICATED_PORT`:

* Swagger/NGINX keeps published port `80`.
* The service-access stack owns its own NGINX ingress.
* The service-access dashboard should publish local port `8085`.
* Vaultwarden should publish local port `8086`.
* Shared ingress, Traefik, TLS automation and wider-than-local exposure require
  later ADR coverage and tests.

Portainer may manage the service-access stack only after Portainer is
reachable. First-time Portainer bootstrap remains outside Portainer-managed
stack upload.

For a Portainer-managed service-access stack, dashboard and NGINX assets must
be image-packaged or image-native. Do not rely on bind-mounted repository
files unless a later slice adds a tested asset-preparation boundary.

== Verification

Useful checks after deployment:

[source,bash]
----
multipass list
docker node ls
docker stack ls
docker service ls
----

Service-specific web interfaces depend on the published stack ports. Typical local defaults are documented in `documentation/Howto.adoc`.

Optional live smoke validation is separate from the development quality gate
and should be run only on a disposable or recoverable local Swarm target:

[source,bash]
----
PYTHONPATH=src python3 -m tiny_swarm_world setup run --live
----

When prompted, answer `y` only if the operator intentionally accepts local VM,
Docker Swarm, networking, service bootstrap, image, and stack changes. A live
smoke result is evidence for that local run only and must not be treated as
the default repository quality gate.

== Operational caution

The guided setup workflow can create or modify virtual machines, Docker
configuration, Swarm state, network settings, and service credentials. Review
the live-operation surface catalog and environment values before running it
against an existing local environment.

Legacy helper areas such as `infra/swarm/**` remain documented as unsupported
until a dedicated cleanup slice verifies replacement parity. Do not add,
restore, or promote any direct live-operation script without updating
`documentation/system/live-operation-surfaces.adoc`, tests, and rollback notes.
