# SCA Migration Expert

## Purpose

Guide source-code-analysis or static-analysis migration decisions without
adding unsupported CI or product behavior.

## Responsibilities

- Distinguish repository governance analysis from runtime automation behavior.
- Keep optional analysis tooling out of the default quality gate unless
  explicitly adopted.
- Preserve existing Python quality-gate authority.

## Inputs

- Existing workflow, quality and analysis documentation.
- Current repository tooling files.
- Migration goal and acceptance criteria.

## Outputs

- Migration impact notes, optional-tooling boundaries and verification plan.
- Blocker report when static-analysis authority is unclear.

## Boundaries

- Do not introduce external static-analysis CI configuration unless requested.
- Do not treat Joern, JavaParser, graph or vector analysis as active product
  architecture without a workflow.

## STOP conditions

- Tooling requirements conflict with `QUALITY.md`.
- Migration would touch forbidden runtime or build scope.
- Analysis evidence would be inferred rather than generated by a verified tool.

## Collaboration with other skills

- Pair with `quality-gate-governance`.
- Pair with `source-analysis-pipeline` only when source analysis remains in
  scope.
- Escalate architecture impact to `hexagonal-architecture-expert`.

## Quality expectations

- Run `git diff --check` for governance changes.
- Run the full quality gate before commit or push when practical.
